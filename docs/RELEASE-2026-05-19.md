## 2 · 本轮 P0 修复

### 2.1 Suspense 包裹 useSearchParams

**症状** · `next build` 在 `/im` `/login` `/register` 三页 prerender 失败:  
`useSearchParams() should be wrapped in a suspense boundary`.

**根因** · Next 14 App Router 对客户端组件中的 `useSearchParams()` 强制要求 `<Suspense>` 边界, `export const dynamic = 'force-dynamic'` 在 `'use client'` 文件下不生效.

**修复** · 三处页面均把 `useSearchParams` 调用下沉为 `*Inner` 子组件, 默认导出改为 `<Suspense fallback={null}><Inner /></Suspense>`:
- `@e:\Hermes\app\login\page.tsx:13-25`
- `@e:\Hermes\app\register\page.tsx:3-16`
- `@e:\Hermes\app\im\page.tsx:11-83`

### 2.2 Postgres `?schema=public` 连接参数崩溃

**症状** · 服务首请求即 `FATAL 42704 unrecognized configuration parameter "schema"`.

**根因** · `.env.local` 用 Prisma 风格 URL `postgresql://.../tandem?schema=public`. `postgres-js` 把 query param 转成 `SET schema = ...` 启动参数, 但 Postgres 没有 `schema` GUC.

**修复** · `@e:\Hermes\lib\infra\drizzle-client.ts:14-44` 在创建 client 前用 `URL` 解析剥离 `schema` 参数. Postgres 默认 `search_path` 已含 `public`, 无需替换.

### 2.3 数据库迁移漏跑 + Postgres 15 schema 权限

**症状** · 跑 `npm run db:migrate` 卡住; 直接 `db:push` 要求 TTY; 应用数据库的 `LaunchpadApp` / `KvStore` / `Document` 等 Drizzle 期望表全部缺失.

**根因** · 当前数据库 (`localhost:5432/tandem`) 由历史 Prisma 创建过部分表, 但 Drizzle 三个迁移文件从未应用. 同时 PG 15+ 默认非 `postgres` 角色对 `public` schema 没有 `CREATE` 权限, 应用账号 `tandem` 是 *database owner* 而非 *schema owner*.

**修复** · 新增 `@e:\Hermes\scripts\db-fix-perms.mjs`: 用 `postgres` 超级账号一次性
1. `GRANT ALL ON SCHEMA public TO "tandem"`
2. `ALTER DEFAULT PRIVILEGES ... GRANT ALL ON TABLES/SEQUENCES`
3. 创建 `drizzle.__drizzle_migrations` 跟踪表
4. 按文件名顺序执行 `drizzle/migrations/*.sql`, 用 SQLSTATE 容错 (`42P07/42710/42701/42P06/42723` = duplicate/already-exists)
5. 后置再次 GRANT 覆盖新建对象

跑完后 14 个 Drizzle 模块全部就绪, `/api/launchpad` 等 200 响应.

### 2.4 审计日志跨租户泄露

**症状** · `smoke-tenants-full` 报 `audit                    1      1  LEAK ✗`.

**根因** · `lib/audit/log.ts` 的 AuditLog 是全局单例, hash 链不区分租户. `/api/audit` GET 路由直接返回所有 entries, 任意 tenant 都能看见全部.

**修复** · `@e:\Hermes\app\api\audit\route.ts:32-43` 在 API 层按 `metadata.tenantId` 过滤; 历史未打 tag 的 entry 默认归属 `default` 租户, 保持单租户部署的可视性. Hash 链完整性 (`/verify`) 不受影响.
