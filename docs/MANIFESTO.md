# 牛马搭子 · Tandem 产品宪章 (Manifesto)

> **「让 AI 做你的搭子, 不让 AI 做你的老板」**
>
> 版本: v1.0
> 最后更新: 2026-05
> 性质: 不可频繁修改的产品根基文档. 所有 PRD / 设计 / 代码决策必须服从本宪章.

---

## 序言: 我们为什么存在

在 AI 真正进入企业之前, 协作工具的设计假设是「人与人沟通」.
飞书、钉钉、企业微信、Microsoft Teams 都是这一时代的代表作.
它们的最小单元是**消息**, 它们的成功度量是**活跃度 (DAU/MAU)**, 它们的 AI 是**消息的辅助生成器**.

在 AI 真正进入企业之后, 这一假设已不再成立.

工作的真正稀缺已经从「信息流通」迁移到「决议产出」.
卷消息、卷加班、卷活跃度, 这些指标对企业未来已没有边际价值.
中国职场最深的痛苦不是「信息不够」, 而是「**为开会而开会, 为汇报而汇报, 为消息而消息**」.

牛马搭子是一个反向回答:

- 我们不让 AI 替员工说话, 我们让 AI 替员工**思考整齐**.
- 我们不让 AI 替员工决策, 我们让 AI 替员工**整理选项**.
- 我们不让 AI 替员工开会, 我们让 AI 替员工**收拾会场**.
- 我们不让 AI 监督员工, 我们让 AI **保护员工免于卷**.

我们写下这份宪章, 作为一切产品决策的不动根基.
**凡与之相违的功能, 不论商业短期收益多大, 一律否决.**

## 文化基座: 四个满意

牛马搭子的产品文化建立在「**四个满意**」之上.

| 满意 | 对应群体 | 我们怎么做 |
|---|---|---|
| 客户满意 | 企业客户 (买单方) | 数据安全 / ROI 透明 / 长期陪伴 |
| 员工满意 | 真实使用者 | 反卷设计 / 心流保护 / 成长可见 |
| 股东满意 | 投资人 / 创始人 | 健康增长 / 健康现金流 / 长期主义 |
| 社会满意 | 行业 / 监管 / 公众 | 合规 / 不做替代型 AI / 推动职场进步 |

四个满意通过**三大核心理念**支撑:

- 🌟 **创造价值** (Create Value) — 为客户、员工、股东、社会都创造可见价值
- 🌟 **赢得尊重** (Earn Respect) — 不靠营销话术, 靠产品本身赢得每一类受众的尊重
- 🌟 **快乐工作** (Work Happily) — 让用户、员工、自己都能在工作中感到快乐

这三大理念是产品所有功能设计的**最终检验标准**.

> **任何功能, 凡与四个满意冲突者 → 一律否决.**

## 不动条款 (18 条)

### 第一条 · 工作的原子单元是「决议」, 不是「消息」

牛马搭子全产品的工作单元是 **Decision Card** (决议卡).

每个 Decision Card 必须包含:
- 关联 KR / TTI
- 上下文链路 (Material 溯源)
- 责任人 (Owner)
- 时间戳
- 影响范围 (其他 KR / 部门)
- 后果回溯指针 (执行后实际效果)

消息只是 Decision Card 的素材, 不是工作本身.

**度量铁律**:
- 会议成功 = 「这个会产生了几张 Decision Card」, 而非「讨论得多热烈」
- 员工成长 = 「TTI 上推动了几张 Decision Card」, 而非「发了多少消息」
- 团队效能 = 「Decision Card 执行率」, 而非「IM 活跃度」

**禁止**: 任何把"消息数 / 在线时长 / 回复速度"作为正向激励的功能.

---

### 第二条 · AI 给 3+1 选项, 不替员工决策

任何 AI 决策辅助场景, 必须呈现:

| 选项 | 来源 | 标签 |
|---|---|---|
| 🅰 SOP 方案 | Memory 层 SOP 库 | 守正 / 制度派 |
| 🅱 推演方案 | 公司 Baseline 模型 | 出奇 / 理性派 |
| 🅲 经验方案 | 案例库 | 借势 / 历史派 |
| 🅳 自创方案 | 员工本人 (强制填"我多看到了什么") | 临场 / 直觉派 |

每个选项必须强制展示:
- 置信度
- 风险评级
- 适用边界
- 关联 KR/TTI 影响

**禁止**:
- 「一键自动给一个最佳建议」(培养橡皮图章, 不培养人才)
- 「AI 输出无依据, 让员工盲信」
- 「Multi-Agent 互相辩论扩散」(违反目标驱动收敛)

**例外**: 常规决策 (SOP 已完整覆盖) 可只给 1 个 SOP 方案 + 一键确认.

---

### 第三条 · 议事室 17 分钟硬上限或升级

议事室 (Convergence Room) 5 步骨架:

```
1. ALIGN     (校准)   2 min  锚定 KR + 同步信息
2. FRAME     (界定)   3 min  问题陈述 + 决策类型识别
3. DIVERGE   (发散)   5 min  AI 给 3+1 选项
4. CONVERGE  (收敛)   4 min  投票表决 / 分歧检测
5. COMMIT    (落地)   3 min  行动项 + Decision Card 生成
─────────────────────────────────────
单议题硬上限 17 分钟
```

**收敛失败时**:
- 分歧 > 阈值 → AI 暴露分歧根源 → 再讨论 1 轮
- 仍分歧 → **自动升级到指定决策人**, AI 不再调解
- 决策人收到的是结构化分歧摘要, 5 分钟即可拍板

**禁止**:
- 1 小时以上的议题会议 (拆成多次议事室)
- 没有 Decision Card 产出的会议
- AI 反复"再想想"无限循环

### 第四条 · KPI → 钱; TTI → 成长 (双轨彻底分离)

```
┌─────────────────┐    ┌─────────────────┐
│  ⚙️ KPI          │    │  🚀 TTI          │
│  刚性管理        │    │  改善牵引        │
├─────────────────┤    ├─────────────────┤
│ 月奖 / 季奖      │    │ 述职材料         │
│ 调薪 / 年终奖    │    │ 晋升评审         │
│ 末位 / PIP       │    │ 高潜识别 (HiPo)  │
│                  │    │ IDP 培养计划     │
└─────────────────┘    └─────────────────┘
       ↓                        ↓
   决定"留得下"             决定"走得远"
```

**双轨独立, 互不污染**:
- TTI 完成情况**不影响**任何形式的金钱回报 (含系数浮动)
- KPI 评分**不直接决定**晋升或高潜身份

**禁止**:
- 把 TTI 完成度乘进绩效公式
- 把 KPI 强制作为高潜入选条件
- 任何"TTI 加成系数"或类似软挂钩设计

这是反 OKR 在中国异化的**核心防御**. 一旦混淆, 全盘崩塌.

---

### 第五条 · KPI 100% 合格; TTI 60-70% 健康

视觉规则铁律:

| 体系 | 颜色 | 完成度规则 |
|---|---|---|
| KPI | 灰 / 蓝 (理性) | < 100% 红色, 100%+ 绿色 |
| TTI | 橙 / 紫 (激励) | 60-80% 绿色, 40-60% 黄色, > 90% **橙警告**: "目标定低了?" |

**KPI 100% 合格的代价 → 设定环节必须严肃**:

KPI 设定时, AI 强制三审视:
1. **历史均值检验**: 设定值 > P75 (过激进) 或 < P50 (过保守) → 警示
2. **客观性检验**: 数据源是否可自动采集? 避免人工填报作弊
3. **可控性检验**: 员工是否真的能控制? 是否有上游依赖卡点?

未通过三审视 → KPI 设定环节阻塞, 不予锁定.

**禁止**:
- KPI 100% 合格但允许"拍脑袋"设定 (会逼出造假)
- TTI 100% 完成不触发"目标定低"提示

---

### 第六条 · 全公司透明 (CEO 的 OKR/TTI 全员可见)

OKR 树最大 **3 层** (公司 → 团队 → 个人).

任何员工可看任何同事的 OKR/TTI, 包含 CEO.

**透明度是 OKR 的灵魂, 不是产品功能**. 牛马搭子默认全透明, 客户后台可关但需高级管理员二次确认.

**例外** (默认私密):
- 个人 KPI 详情 (上司 + 本人 + HR 可见)
- 述职材料中员工本人不愿公开部分
- 红区会议 Decision Card (合规所需)

---

### 第七条 · Material ≠ Memory (材料层 vs 记忆层)

四层知识架构:

```
Layer 1: ORIGINS    (录像/原始消息)        ← 仅当事人可见
              ↓ 自动
Layer 2: MATERIALS  (纪要/Decision Card)   ← 全员可写可查
              ↓ 严肃签批 ⚖️
Layer 3: MEMORY     (SOP/案例/红线/价值观) ← 签批后入, 全员引用
              ↓ 季度训练 🧠
Layer 4: BASELINE   (公司大模型权重)
```

**核心区分**:
- Material 描述「事实如何」(描述性)
- Memory 描述「我们公司认为应该如何」(规范性)

**Memory 的特殊地位**:
Memory 是**三件事的核心素材**:
1. 企业级集体记忆
2. 个人 AI (拿捏老板) 的成长养料
3. 3+1 决策中"推演方案"的训练源头

因此 Memory 的**写入和降级**都必须经严肃流程, **不允许任何自动化判定**.

**禁止**:
- Material 直接喂给 Baseline 训练 (会污染基线)
- 没有标注层级的混合知识库
- 把会议纪要等同于公司决议
- 让 AI 凭时间或引用率自动归档 Memory

### 第八条 · 公司记忆必须经签批 (防基线漂移)

#### 8.1 升级签批闸门 (Material → Memory)

| 级别 | 签批人 | SLA | 适用范围 |
|---|---|---|---|
| Lv1 团队级 | 团队 Leader + 知识治理官 | 3 工作日 | 团队 SOP / 小型案例 |
| Lv2 部门级 | 部门 Leader + 治理官 + KR 关联人 | 5 工作日 | 部门 SOP / 跨团队案例 |
| Lv3 公司级 | C-level 集体 | 14 工作日 | 红线 / 价值观 / 战略叙事 |

**逾期机制**: 所有等级逾期自动 escalate +1 级
- Lv1 逾期 → 升 Lv2 处理
- Lv2 逾期 → 升 Lv3 处理
- Lv3 逾期 → 系统通知 CEO + 治理委员会

未通过签批 → 留在 Material 层, 仍可被检索引用, 但不构成基线训练数据.

#### 8.2 降级管理评估 (Memory → Material/归档)

Memory 中已存在的条目, **不允许基于时间或低引用率自动归档**.

降级流程 (与升级同等严肃):
1. **AI 主动通知治理官**: "SOP-042 引用率连续 3 季度低于均值 30%, 建议评估"
2. **治理官评估**: 召开评审小组 (类似签批流程)
3. **决议产出**: 保留 / 修订 / 归档 / 标注 "仅历史参考" 之一
4. **决议记录**: 进入 Memory 自身的版本历史

**禁止**:
- 任何"自动判定为公司记忆"的逻辑
- 任何"基于时间自动归档 Memory"的逻辑
- 治理官由直接业务 Leader 兼任 (避免裁判员兼运动员)

### 第九条 · 分身代参必须显式标识 (反 AI 欺诈)

#### 9.1 显式标识铁律

分身入会必须:
- 强制水印: 音频自动加 "AI 代理" 标识 (员工不可关闭)
- 视频上叠加 "AI 分身代参中" 字样
- 会议邀请方有权拒绝分身参会 (主持人开关)

#### 9.2 红区会议禁用分身

- 客户 / 合作伙伴 / 供应商会议
- 招聘面试 / 绩效面谈
- 战略 / C-level / 董事会
- 法务 / 合规 / 审计 / 财务

跨公司会议代参 → 必须双方**书面同意** + 双倍水印强制.

#### 9.3 控制中枢: 事后纪要确认

分身代参的核心治理点是**员工事后对会议纪要的强制确认**:

- 会议结束 → 自动生成纪要 + 决议清单 → 推送员工
- 否决窗口: **24 小时**
- 员工可一键否决任何分身决议或代发言
- 否决后系统自动通知会议参与方"该决议被员工修正"

**这一确认机制对所有 SKU 开放** (Free / Personal / Enterprise), 不基于商业等级 gate.
唯一区别: **音色克隆**仅企业版默认采集 (Free / Personal 用 AI 通用音色).

**禁止**:
- 静默代参 (对方不知是 AI)
- 一次授权终身代参
- 在红区会议白名单中加入任何例外
- 用 SKU 屏蔽分身代参的"事后确认"权利

### 第十条 · 9 宫格人才矩阵 (KPI × TTI)

```
                          KPI (实绩) →
              不达标 (<100%)   达标 (100%)    超出 (>100%)
         ┌─────────────────┬──────────────┬─────────────────┐
   突出  │ ❓ 待开发高潜    │ 💎 高潜稳健   │ ⭐ 核心接班人    │
  TTI    │ → 调岗/给机会    │ → 加速培养    │ → 高曝光项目     │
  (潜力) ├─────────────────┼──────────────┼─────────────────┤
   ↑  中 │ ⚠️ 方向错位     │ 🟢 中坚骨干   │ 🚀 强力贡献者    │
         │ → 1:1 校准       │ → 稳定激励    │ → 提拔候选       │
         ├─────────────────┼──────────────┼─────────────────┤
   低    │ 🔴 末位 PIP      │ ✅ 稳定老将   │ 🎯 技术专家      │
         │ → 警示或淘汰     │ → 维护现状    │ → 专家通道       │
         └─────────────────┴──────────────┴─────────────────┘
```

每位员工每季度自动归格. AI 给经理推荐每格的具体管理动作.

**末位淘汰仅基于 X 轴 (KPI), 不基于 Y 轴 (TTI)**.
TTI 不达成不会触发任何惩罚.

Calibration 校准会用盲打分 (屏蔽姓名), 去除偏见.

**禁止**:
- 末位强制分布 (10/20/70 stack ranking)
- 用 TTI 完成度直接淘汰员工
- 用 KPI 完成度直接判定高潜

### 第十一条 · 反对消息黏性, 拥抱异步聚合 (心流神圣)

消息默认进 AI Digest, **每天 2-3 次集中处理**.
紧急通道 (@紧急 / 老板加急) 严格限额 (每人每天 ≤ 3 次).
让员工每天有 **4-6 小时不被打断的心流时间**.

#### 11.1 已读机制的边界

「已读」不是禁忌, **滥用「已读」才是禁忌**.

**允许的已读机制 (决议型, 必做)**:

- ✅ 会议邀请的明确回复 (本人 / 分身 / 拒绝)
- ✅ Decision Card 的阅签确认
- ✅ 紧急任务的接收确认
- ✅ 签批申请的处理状态 (已读/在审/已批)
- ✅ TTI/KPI 关键节点的承认确认

**禁止的已读机制 (焦虑型, 不做)**:

- ❌ IM 普通聊天的秒级红点
- ❌ "X 分钟未回复"自动提醒
- ❌ 上司可看下属"几点几分读了消息"的监控视图
- ❌ "已读未回"的红字标记 (制造对方压力)

核心区分: **决议型已读对工作有意义; 焦虑型已读只制造压力**.

#### 11.2 禁止任何"提高活跃度"的产品激励

- ❌ 任何 DAU/MAU 指标的产品激励
- ❌ Push 通知滥发
- ❌ 排行榜诱导员工"卷消息"
- ❌ 在线时长奖励

我们不靠黏性赚钱. 我们靠员工成长 + 老板心安赚钱.

---

### 第十二条 · 末位机制基于绝对 KPI, 不基于相对排名

#### 12.1 末位机制保留, 但有边界

中国企业管理需要末位机制, 我们提供. 但有铁律.

**允许的末位机制 (绝对型, 必做)**:

- ✅ KPI 连续 N 季度不达标 → PIP 改进计划触发
- ✅ KPI 严重低于岗位说明书底线 → 系统预警
- ✅ 客观数据 (如客诉率、错误率) 触发预警
- ✅ Calibration 校准会综合评估末位候选
- ✅ 9 宫格 🔴 末位象限自动识别

**禁止的末位机制 (强制分布, 不做)**:

- ❌ "团队必须淘汰末位 10%" 强制比例
- ❌ Stack Ranking (10/20/70) 强制分布配置
- ❌ "上有政策下必须凑数" 的强制项
- ❌ 末位与 TTI 完成度挂钩 (TTI 不达成永不触发末位)
- ❌ 跨团队相对排名决定淘汰

#### 12.2 为什么不做强制分布

GE 时代的强制分布已被现代管理学公认有害:

- 团队整体优秀也必须淘汰一人 → 反生产力
- 制造内部恶性竞争 → 破坏协作
- Microsoft 已废弃 stack ranking 多年, 业绩反升

我们提供末位机制是为**保护企业和团队**, 不是为**制造内卷**.

**禁止**: 在产品后台设置任何"必须有 X% 末位"的强制分布项.

### 第十三条 · 数据归公司, 尊严归员工

牛马搭子的数据归属遵循**一明一暗两条原则**.

#### 13.1 数据归公司 (明面契约)

员工在岗期间, 通过牛马搭子产生的**所有工作数据归企业客户所有**:

- OKR / KPI / TTI 数据
- Decision Card 与决议历史
- 会议纪要与录像 (Origins / Materials)
- Memory 层 SOP / 案例 / 红线
- 客户信息 / 业务数据
- 个体分身 (拿捏老板) 的工作上下文与权重

员工离职 → 上述数据**全部留在企业**, 由企业按合规要求处置.

这是中国 To B 商业现实, 也是企业客户买单的合理诉求.

#### 13.2 尊严归员工 (暗面铁律)

虽然数据归公司, **员工的尊严不可被产品侵犯**:

- 不可监控员工的"在线时长 / 输入速度 / 屏幕活动"
- 不可在员工不知情的情况下采集生物特征 (声纹/人脸)
- 不可强制员工在 8 小时外响应消息
- 不可基于性别 / 年龄 / 婚育做差别化建议
- 不可让 AI 替员工承担情绪压力 (例如代员工被骂)
- 不可在员工离职后抹黑员工档案
- 不可在公开场合披露员工的 9 宫格末位身份

#### 13.3 员工的合理获得物

虽然数据归公司, 员工离职时仍可获得:

- 个人成长报告 (PDF 摘要, 不含公司机密)
- 述职记录摘要 (员工本人版本)
- 拿捏老板使用统计 (匿名化, 用于个人简历)

这不是数据可携, 是**对劳动尊严的最低保障**, 也是「员工满意」基座的产品体现.

**禁止**:

- 任何"监控员工日常状态"的功能
- 任何"用 AI 替代员工承担情绪压力"的功能
- 任何剥夺员工最低尊严保障的设计

### 第十四条 · 知识治理官独立角色 (防腐败)

Knowledge Steward 是产品 RBAC 中的**独立角色**, 不可由以下兼任:
- 直接业务 Leader (避免裁判兼运动员)
- HR (避免与绩效混淆)

允许由以下兼任 (中小公司):
- 战略部门负责人
- 总裁办 / Chief of Staff
- 资深产品/技术专家 (兼任比例 ≤ 30%)

大客户可设专岗.

治理官工作流:
- 审核 Material → Memory 升级申请 (第八条 8.1)
- 主持 Memory → 归档/修订评估 (第八条 8.2)
- 维护 SOP / 案例 / 红线体系健康度
- 季度审计 Memory 漂移与失效
- 主持年初战略部署 Workshop

治理官有专属 AI Co-pilot, 自动标注矛盾 / 过时 / 引用统计.

### 第十五条 · AI 助员工成长, 不替员工劳动

牛马搭子的所有 AI 输出, 必须满足两条铁律:

1. **可被员工否决** (24h 内否决窗口)
2. **每次输出都让员工更聪明一点** (展示推演过程, 不只给结论)

**被禁止的 AI 形态**:
- ❌ 「一键自动完成所有工作, 员工只需点确认」 → 是替代不是辅助
- ❌ 「无解释黑盒输出」 → 员工无法学习, 只能盲信
- ❌ 「连续 N 周替员工做同一类决策」 → 员工技能退化检测 + 主动降级权限

**我们的口号**:

> **「下班早一小时」是表象, 「员工成长更快」才是承诺.**

---

### 第十六条 · LLM 可热插拔, TAF 不可妥协

基座大模型 (LLM) 是**燃料**, 不是引擎. **引擎是 TAF (Tandem Agent Framework).**

**铁律**:

1. **LLM 抽象层**: 通过 OpenAI 兼容接口标准化, 任何模型一行代码切换 (DeepSeek / Qwen / Doubao / Kimi / GLM / Hermes / 国际模型)
2. **协议层固化**: 3+1 决策 / 议事室 5 步 / Decision Card / Persona / Memory 在 TAF 协议层定义, 不绑任何具体模型
3. **Hermes Watch 机制**: 持续跟随 Nous Research / HuggingFace / Cline 等开源生态进化, 月报评估
4. **协议优先于权重**: Hermes Function Calling 格式作为 TAF Layer 3 标准, vLLM / Together / Fireworks 原生兼容, 国产模型 prompt 适配
5. **多模型路由**: 按场景 (推理/工具/长上下文/高频) + 成本 + 失败回退自动选择

**反例**:

- ❌ 把业务逻辑写进单一模型的 fine-tune (绑死, 进化困难)
- ❌ 自训基座大模型 (创业期资源浪费)
- ❌ 锁定单一 LLM 厂商 (议价能力丧失)

**口号**:

> **「不养自己的模型, 养自己的协议; 不抄 Hermes 的代码, 接 Hermes 的进化.」**

---

### 第十七条 · 我们做民企的牛马搭子

**客户边界一刀切**:

```
✅ 目标客户:
   • 互联网 / SaaS / 消费 / 教育 / 创意 / 跨境电商 / 文化娱乐
   • 50-3000 人民营企业 (sweet spot 200-1000)
   • 推崇 OKR / 反卷 / 远程友好 / AI 友好
   • 决策人: CEO / COO / CHRO / 创始人
   • 在华外资企业

❌ 不进入市场:
   • 中央 / 地方政府机关
   • 国有企业 (含央企 / 地方国企)
   • 银行 / 证券 / 保险等金融机构
   • 涉密单位 / 军工 / 公检法
```

**为什么**:

- 民企能接受我们的反卷价值观 (国企体制矛盾大)
- 民企能接受 LLM 多元化 (政企强制信创 / 国产化)
- 民企决策快 (政企招标周期长)
- 民企认 PLG (员工自下而上, 政企必须 SLG)

**反例 (我们不做)**:

- ❌ 不做信创版 / 涉密版 / 国密算法版
- ❌ 不进政企招标 / 国资云 / 行业云
- ❌ 不为政企妥协价值观 (如关绩效不挂钩开关默认值)
- ❌ 不做强制分布功能 (即使大客户要求)

**长期视野**:

民企做出标杆后, 政企自然会学习. 我们不主动入场, 但**欢迎政企开明派来学**. 我们不为他们做特殊版本.

**口号**:

> **「我们做民企的牛马搭子. 让民企员工先翻身. 政企以后愿意学, 欢迎学.」**

### 第十八条 · OSS 借力 + 自建思考层

牛马搭子主体**自建完整产品**, 但底座能力**全面借力 OSS + AI 加速开发**, 以 12-14 月 V1 GA 为目标.

**铁律**:

```
✅ 自建 (核心差异化, 100% 投入):
   • OKR/TTI + Decision Card + 9 宫格
   • 议事室 5 步 + 3+1 决策
   • 拿捏老板 (Persona Engine)
   • TAF (Tandem Agent Framework)
   • 四层知识架构 + Steward
   • Inbox 聚合层

✅ 自建 (复用 Hermes 驱动 + PG, 不引第二套 stack):
   • IM 消息 + 群聊       → 自建 (Hermes runtime + Next.js + PG)
                            理由: PoC 已跑通差异化层 (spawn-room / @persona), 引 Rocket.Chat 等于
                            (1) 多一套 Mongo 副本集运维, (2) 上游分支同步成本, (3) SSO 桥接复杂度,
                            (4) 哲学上违反 "Hermes 是底座" — IM 应作为 Hermes 上的应用, 不是平行栈.
   • 组织架构 + 通讯录      → 自建 (Hermes 基础 + 企微/钉钉初始导入)
   • 邮件                  → 腾讯企邮 + Outlook API 调用

✅ 自建 (借力 OSS, fork 改造 — 仅当 PoC 自建成本 ≫ fork 成本时启动):
   • 日历                  → Cal.com fork (V1 后期评估)
   • 文档协作              → Yjs + Tiptap (Doc) + Univer (Sheet)
   • 云盘 / 文件存储         → MinIO + 自建 UI

✅ 寄生 (调 API):
   • 音视频会议           → 腾讯会议 (海外 V3 加 LiveKit)

❌ 永不做:
   • CRM / 客户管理 (走 HubSpot / 纷享销客 对接)
   • OA / 审批 / 考勤 / 打卡 (走 企微/钉钉/HRIS 对接)
   • 外勤 / 巡店 / GPS
   • 印章 / 电子签 (走专业服务商)
```

**为什么 OSS 借力**:

1. **不重造轮子**: IM/邮件/日历/云盘 有 10+ 年成熟 OSS, 重写是资源浪费
2. **AI 加速改造**: Cascade/Cursor 助我们快速 fork + 改变 + 集成
3. **产权明确**: MIT / Apache 2 协议零风险商用, AGPL 需法务 review
4. **上面坚持自建**: 上层思考层 (TAF/OKR/议事室) 必须自建, 这是护城河

**反例**:

- ❌ 不为了"完全自主"重写 IM (24 月 vs 1 月 fork)
- ❌ 不重写 SMTP/IMAP 邮件服务器 (腾讯企邮已是世界一流)
- ❌ 不重写企业 CRM (专业 CRM 商业生态成熟)
- ❌ 不赋能 OA 审批/考勤 (不是我们场景)

**唯一根据**: 如某个 OSS 不能满足中国民企场景, 由 CTO + 产品共同决议是否改为自建.

**口号**:

> **「上面思考层是我们的, 下面底座是社区的. 不为轮子舍本, 不为服从妥协.」**

## 与三巨头的本质差别 (一句话叙事)

| 维度 | 飞书/钉钉/企微 | 牛马搭子 (Tandem) |
|---|---|---|
| 工作单元 | 消息 | 决议卡 (Decision Card) |
| 成功度量 | 活跃度 (DAU/MAU) | 决议执行率 |
| AI 形态 | 消息生成器 | 决策结构化器 |
| 员工视角 | 被监督的对象 | 被赋能的主体 |
| 进入路径 | 公司买单 (SLG) | 员工自下而上 (PLG) |
| 数据归属 | 全部归公司, 员工无尊严保障 | 数据归公司 + 员工尊严铁律保障 |
| 时间观 | 鼓励实时在线 | 鼓励异步心流 |
| 知识管理 | 单层 Wiki (死) | 四层架构 (活) |
| 绩效管理 | OKR 直接挂薪资 | KPI/TTI 双轨 |
| 成长承诺 | 没有 | 9 宫格 + IDP + 拿捏老板辅助 |
| 会议参与 | 必须本人到场 | 本人 / 分身可选 (微信式) |
| 文化基座 | DAU/MAU | 四个满意 (客户/员工/股东/社会) |

**一句话**:

> 「飞书让你**永远在线**. 牛马搭子让你**该在的时候在, 不该在的时候有人替你**.」

## 附录 B: 5 问产品决策检验

任何新功能上线前, 用以下 5 问自检:

1. ❓ 这个功能是让员工**离决议更近, 还是离消息更近**?
2. ❓ 这个功能让 AI 是**辅助员工, 还是替代员工**?
3. ❓ 这个功能让 KPI / TTI 双轨**保持分离, 还是混淆**?
4. ❓ 这个功能让员工**心流时间增加, 还是被打断更多**?
5. ❓ 这个功能尊重**员工是主体, 还是把员工当资源**?

**5 个问题任一答案是"前者"** → 通过
**任一答案是"后者"** → 设计不合格, 重新做
