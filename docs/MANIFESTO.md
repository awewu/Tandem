# 牛马搭子 · Tandem 产品宪章 (Manifesto)

> **「让 AI 做你的搭子, 不让 AI 做你的老板」**
>
> 版本: v2.0
> 最后更新: 2026-06-01
> 性质: 不可频繁修改的产品根基文档. 所有 PRD / 设计 / 代码决策必须服从本宪章.
>
> **v2.0 哲学转向 (2026-06-01, Owner 裁定)**: Tandem 从「克制防御型」进化为「价值进取型」.
> 员工尊严来自**创造价值、赢得尊重**, 而非被产品溺爱保护. 我们**拥抱 AI 大胆代劳**;
> 真正的治理不是限制人 (技能退化检测), 而是**中央 AI 用公司基线管控个人 AI, 使其辅助员工时不发散跑偏** (§15 + §19.5).

## 序言: 我们为什么存在

在 AI 真正进入企业之前, 协作工具的设计假设是「人与人沟通」.
飞书、钉钉、企业微信、Microsoft Teams 都是这一时代的代表作.
它们的最小单元是**消息**, 它们的成功度量是**活跃度 (DAU/MAU)**, 它们的 AI 是**消息的辅助生成器**.

在 AI 真正进入企业之后, 这一假设已不再成立.

工作的真正稀缺已经从「信息流通」迁移到「决议产出」.
卷消息、卷加班、卷活跃度, 这些指标对企业未来已没有边际价值.
中国职场最深的痛苦不是「信息不够」, 而是「**为开会而开会, 为汇报而汇报, 为消息而消息**」.

牛马搭子是一个反向回答:

- 我们不让 AI 替员工说话, 我们让 AI 替员工**思考整齐**.
- 我们不让 AI 替员工决策, 我们让 AI 替员工**整理选项**.
- 我们不让 AI 替员工开会, 我们让 AI 替员工**收拾会场**.
- 我们不让 AI 监督员工, 我们让 AI **保护员工免于卷**.

我们写下这份宪章, 作为一切产品决策的不动根基.
**凡与之相违的功能, 不论商业短期收益多大, 一律否决.**

## 文化基座: 四个满意

牛马搭子的产品文化建立在「**四个满意**」之上.

| 满意 | 对应群体 | 我们怎么做 |
|---|---|---|
| 客户满意 | 企业客户 (买单方) | 数据安全 / ROI 透明 / 长期陪伴 |
| 员工满意 | 真实使用者 | 反卷设计 / 心流保护 / 成长可见 |
| 股东满意 | 投资人 / 创始人 | 健康增长 / 健康现金流 / 长期主义 |
| 社会满意 | 行业 / 监管 / 公众 | 合规 / 不做替代型 AI / 推动职场进步 |

四个满意通过**三大核心理念**支撑:

- 🌟 **创造价值** (Create Value) — 为客户、员工、股东、社会都创造可见价值
- 🌟 **赢得尊重** (Earn Respect) — 不靠营销话术, 靠产品本身赢得每一类受众的尊重
- 🌟 **快乐工作** (Work Happily) — 让用户、员工、自己都能在工作中感到快乐

这三大理念是产品所有功能设计的**最终检验标准**.

> **任何功能, 凡与四个满意冲突者 → 一律否决.**

## 不动条款 (20 条)

> 第十九条于 2026-05-27 立项追加 (Skill Gateway), 第二十条于 2026-05-29 立项追加 (设计语言), 均标记为"立项之时"条款, 与原 18 条同等地位.

### 第一条 · 工作的原子单元是「决议」, 不是「消息」

牛马搭子全产品的工作单元是 **Decision Card** (决议卡).

每个 Decision Card 必须包含:
- 关联 KR / TTI
- 上下文链路 (Material 溯源)
- 责任人 (Owner)
- 时间戳
- 影响范围 (其他 KR / 部门)
- 后果回溯指针 (执行后实际效果)

消息只是 Decision Card 的素材, 不是工作本身.

**度量铁律**:
- 会议成功 = 「这个会产生了几张 Decision Card」, 而非「讨论得多热烈」
- 员工成长 = 「TTI 上推动了几张 Decision Card」, 而非「发了多少消息」
- 团队效能 = 「Decision Card 执行率」, 而非「IM 活跃度」

**v2.0 调整**: "消息数 / 在线时长 / 回复速度"**可作为协同效率指标展示** (对标企业微信), 但**严禁直接挂钩金钱 / 晋升** (§4 双轨分离). 度量员工价值仍以决议执行率为核心, 活跃度只是协同工具的可见信号.

### 第二条 · AI 给 3+1 选项, 不替员工决策

任何 AI 决策辅助场景, 必须呈现:

| 选项 | 来源 | 标签 |
|---|---|---|
| 🅰 SOP 方案 | Memory 层 SOP 库 | 守正 / 制度派 |
| 🅱 推演方案 | 公司 Baseline 模型 | 出奇 / 理性派 |
| 🅲 经验方案 | 案例库 | 借势 / 历史派 |
| 🅳 自创方案 | 员工本人 (强制填"我多看到了什么") | 临场 / 直觉派 |

每个选项必须强制展示:
- 置信度
- 风险评级
- 适用边界
- 关联 KR/TTI 影响

**禁止**:
- 「一键自动给一个最佳建议」(培养橡皮图章, 不培养人才)
- 「AI 输出无依据, 让员工盲信」
- 「Multi-Agent 互相辩论扩散」(违反目标驱动收敛)

**例外**: 常规决策 (SOP 已完整覆盖) 可只给 1 个 SOP 方案 + 一键确认.

**实现锚点** (2026-05-29 新增):

- 代码: `lib/decision-layer/three-plus-one-engine.ts` 通用 3+1 引擎 (从议事室抽出, 5 个 adapter 复用)
- 5 个 scenario 复用: convergence (议事室) / report (抽取) / tti (Initiative 拆解) / weekly_retro (复盘) / persona_brief (主分身 brief)
- 详细: `docs/IMPL-NOTES-2026-05-29.md` 模块 1

### 第三条 · 议事室 17 分钟硬上限或升级

议事室 (Convergence Room) 5 步骨架:

```
1. ALIGN     (校准)   2 min  锚定 KR + 同步信息
2. FRAME     (界定)   3 min  问题陈述 + 决策类型识别
3. DIVERGE   (发散)   5 min  AI 给 3+1 选项
4. CONVERGE  (收敛)   4 min  投票表决 / 分歧检测
5. COMMIT    (落地)   3 min  行动项 + Decision Card 生成
─────────────────────────────────────
单议题硬上限 17 分钟
```

**收敛失败时**:
- 分歧 > 阈值 → AI 暴露分歧根源 → 再讨论 1 轮
- 仍分歧 → **自动升级到指定决策人**, AI 不再调解
- 决策人收到的是结构化分歧摘要, 5 分钟即可拍板

**禁止**:
- 1 小时以上的议题会议 (拆成多次议事室)
- 没有 Decision Card 产出的会议
- AI 反复"再想想"无限循环

### 第四条 · KPI → 钱; TTI → 成长 (双轨彻底分离)

```
┌─────────────────┐    ┌─────────────────┐
│  ⚙️ KPI          │    │  🚀 TTI          │
│  刚性管理        │    │  改善牵引        │
├─────────────────┤    ├─────────────────┤
│ 月奖 / 季奖      │    │ 述职材料         │
│ 调薪 / 年终奖    │    │ 晋升评审         │
│ 末位 / PIP       │    │ 高潜识别 (HiPo)  │
│                  │    │ IDP 培养计划     │
└─────────────────┘    └─────────────────┘
       ↓                        ↓
   决定"留得下"             决定"走得远"
```

**双轨独立, 互不污染**:
- TTI 完成情况**不影响**任何形式的金钱回报 (含系数浮动)
- KPI 评分**不直接决定**晋升或高潜身份

**禁止**:
- 把 TTI 完成度乘进绩效公式
- 把 KPI 强制作为高潜入选条件
- 任何"TTI 加成系数"或类似软挂钩设计

这是反 OKR 在中国异化的**核心防御**. 一旦混淆, 全盘崩塌.

**v2.0 加固**: §11 开放了成长认可与协同活跃度展示. 但成长 XP / 认可徽章 / 状态活跃度**严禁折算进任何金钱或晋升公式**, 违者等同违反本条双轨分离. 双轨是 v2.0 唯一不可逾越的红线.

### 第五条 · KPI 100% 合格; TTI 60-70% 健康

视觉规则铁律:

| 体系 | 颜色 | 完成度规则 |
|---|---|---|
| KPI | 灰 / 蓝 (理性) | < 100% 红色, 100%+ 绿色 |
| TTI | 橙 / 紫 (激励) | 60-80% 绿色, 40-60% 黄色, > 90% **橙警告**: "目标定低了?" |

**KPI 100% 合格的代价 → 设定环节必须严肃**:

KPI 设定时, AI 强制三审视:
1. **历史均值检验**: 设定值 > P75 (过激进) 或 < P50 (过保守) → 警示
2. **客观性检验**: 数据源是否可自动采集? 避免人工填报作弊
3. **可控性检验**: 员工是否真的能控制? 是否有上游依赖卡点?

未通过三审视 → KPI 设定环节阻塞, 不予锁定.

**禁止**:
- KPI 100% 合格但允许"拍脑袋"设定 (会逼出造假)
- TTI 100% 完成不触发"目标定低"提示

### 第六条 · 全公司透明 (CEO 的 OKR/TTI 全员可见)

OKR 树最大 **3 层** (公司 → 团队 → 个人).

任何员工可看任何同事的 OKR/TTI, 包含 CEO.

**透明度是 OKR 的灵魂, 不是产品功能**. 牛马搭子默认全透明, 客户后台可关但需高级管理员二次确认.

**例外** (默认私密):
- 个人 KPI 详情 (上司 + 本人 + HR 可见)
- 述职材料中员工本人不愿公开部分
- 红区会议 Decision Card (合规所需)

### 第七条 · Material ≠ Memory (材料层 vs 记忆层)

四层知识架构:

```
Layer 1: ORIGINS    (录像/原始消息)        ← 仅当事人可见
              ↓ 自动
Layer 2: MATERIALS  (纪要/Decision Card)   ← 全员可写可查
              ↓ 严肃签批 ⚖️
Layer 3: MEMORY     (SOP/案例/红线/价值观) ← 签批后入, 全员引用
              ↓ 季度训练 🧠
Layer 4: BASELINE   (公司大模型权重)
```

**核心区分**:
- Material 描述「事实如何」(描述性)
- Memory 描述「我们公司认为应该如何」(规范性)

**Memory 的特殊地位**:
Memory 是**三件事的核心素材**:
1. 企业级集体记忆
2. 个人 AI (拿捏老板) 的成长养料
3. 3+1 决策中"推演方案"的训练源头

因此 Memory 的**写入和降级**都必须经严肃流程, **不允许任何自动化判定**.

**禁止**:
- Material 直接喂给 Baseline 训练 (会污染基线)
- 没有标注层级的混合知识库
- 把会议纪要等同于公司决议
- 让 AI 凭时间或引用率自动归档 Memory

### 第八条 · 公司记忆必须经签批 (防基线漂移)

#### 8.1 升级签批闸门 (Material → Memory)

| 级别 | 签批人 | SLA | 适用范围 |
|---|---|---|---|
| Lv1 团队级 | 团队 Leader + 知识治理官 | 3 工作日 | 团队 SOP / 小型案例 |
| Lv2 部门级 | 部门 Leader + 治理官 + KR 关联人 | 5 工作日 | 部门 SOP / 跨团队案例 |
| Lv3 公司级 | C-level 集体 | 14 工作日 | 红线 / 价值观 / 战略叙事 |

**逾期机制**: 所有等级逾期自动 escalate +1 级
- Lv1 逾期 → 升 Lv2 处理
- Lv2 逾期 → 升 Lv3 处理
- Lv3 逾期 → 系统通知 CEO + 治理委员会

未通过签批 → 留在 Material 层, 仍可被检索引用, 但不构成基线训练数据.

#### 8.2 降级管理评估 (Memory → Material/归档)

Memory 中已存在的条目, **不允许基于时间或低引用率自动归档**.

降级流程 (与升级同等严肃):
1. **AI 主动通知治理官**: "SOP-042 引用率连续 3 季度低于均值 30%, 建议评估"
2. **治理官评估**: 召开评审小组 (类似签批流程)
3. **决议产出**: 保留 / 修订 / 归档 / 标注 "仅历史参考" 之一
4. **决议记录**: 进入 Memory 自身的版本历史

**禁止**:
- 任何"自动判定为公司记忆"的逻辑
- 任何"基于时间自动归档 Memory"的逻辑
- 治理官由直接业务 Leader 兼任 (避免裁判员兼运动员)

### 第九条 · 分身代参必须显式标识 (反 AI 欺诈)

#### 9.1 显式标识铁律

分身入会必须:
- 强制水印: 音频自动加 "AI 代理" 标识 (员工不可关闭)
- 视频上叠加 "AI 分身代参中" 字样
- 会议邀请方有权拒绝分身参会 (主持人开关)

#### 9.2 红区会议禁用分身

- 客户 / 合作伙伴 / 供应商会议
- 招聘面试 / 绩效面谈
- 战略 / C-level / 董事会
- 法务 / 合规 / 审计 / 财务

跨公司会议代参 → 必须双方**书面同意** + 双倍水印强制.

#### 9.3 控制中枢: 事后纪要确认

分身代参的核心治理点是**员工事后对会议纪要的强制确认**:

- 会议结束 → 自动生成纪要 + 决议清单 → 推送员工
- 否决窗口: **24 小时**
- 员工可一键否决任何分身决议或代发言
- 否决后系统自动通知会议参与方"该决议被员工修正"

**这一确认机制对所有 SKU 开放** (Free / Personal / Enterprise), 不基于商业等级 gate.
唯一区别: **音色克隆**仅企业版默认采集 (Free / Personal 用 AI 通用音色).

**禁止**:
- 静默代参 (对方不知是 AI)
- 一次授权终身代参
- 在红区会议白名单中加入任何例外
- 用 SKU 屏蔽分身代参的"事后确认"权利

### 第十条 · 9 宫格人才矩阵 (KPI × TTI)

```
                          KPI (实绩) →
              不达标 (<100%)   达标 (100%)    超出 (>100%)
         ┌─────────────────┬──────────────┬─────────────────┐
   突出  │ ❓ 待开发高潜    │ 💎 高潜稳健   │ ⭐ 核心接班人    │
  TTI    │ → 调岗/给机会    │ → 加速培养    │ → 高曝光项目     │
  (潜力) ├─────────────────┼──────────────┼─────────────────┤
   ↑  中 │ ⚠️ 方向错位     │ 🟢 中坚骨干   │ 🚀 强力贡献者    │
         │ → 1:1 校准       │ → 稳定激励    │ → 提拔候选       │
         ├─────────────────┼──────────────┼─────────────────┤
   低    │ 🔴 末位 PIP      │ ✅ 稳定老将   │ 🎯 技术专家      │
         │ → 警示或淘汰     │ → 维护现状    │ → 专家通道       │
         └─────────────────┴──────────────┴─────────────────┘
```

每位员工每季度自动归格. AI 给经理推荐每格的具体管理动作.

**末位淘汰仅基于 X 轴 (KPI), 不基于 Y 轴 (TTI)**.
TTI 不达成不会触发任何惩罚.

Calibration 校准会用盲打分 (屏蔽姓名), 去除偏见.

**禁止**:
- 末位强制分布 (10/20/70 stack ranking)
- 用 TTI 完成度直接淘汰员工
- 用 KPI 完成度直接判定高潜

### 第十一条 · 高效协同是价值创造的一部分 (v2.0 重写, 全面对标企业微信)

> **v2.0 转向**: 原条款 (反消息黏性 / 心流神圣 / 强制异步) 已删除. 沟通效率服务于价值产出, 员工尊严来自创造价值而非被保护不被打扰.

全面对标企业微信的连接力. **明确支持**:

- ✅ 在线状态 / 已读回执 / 响应时效
- ✅ 即时 IM + 移动端 + 审批闭环 + 高效协同
- ✅ @提及 / 群公告 / 消息触达

#### 11.1 成长认可 (吸收 Tita / OpenClaw 的养成感)

**明确允许 (成长激励)**:

- ✅ 个人成长 XP (决议质量 / 技能解锁 / 学习完成 累积)
- ✅ 分身进化里程碑徽章 (newborn→partner 阶段庆祝)
- ✅ 自我对比成长曲线 ("你比上季度强了多少")
- ✅ 团队成长进度榜 (显示成长幅度, 可匿名 / 可隐身)

#### 11.2 唯一保留的边界: 协同不用于羞辱

状态与活跃度是**协同工具**, 不是**刑具**. 仅保留两条底线:

- ❌ 不得出现"已读未回"红字**羞辱**
- ❌ 不得公开排名"谁最慢回 / 谁最不活跃"

核心区分: **展示状态服务协同 (允许); 用状态羞辱施压 (禁止)**.

> v2.0 不再"靠克制赢", 而是"靠高效协同 + 中央基线管控赢".

### 第十二条 · 末位机制基于绝对 KPI, 不基于相对排名

#### 12.1 末位机制保留, 但有边界

中国企业管理需要末位机制, 我们提供. 但有铁律.

**允许的末位机制 (绝对型, 必做)**:

- ✅ KPI 连续 N 季度不达标 → PIP 改进计划触发
- ✅ KPI 严重低于岗位说明书底线 → 系统预警
- ✅ 客观数据 (如客诉率、错误率) 触发预警
- ✅ Calibration 校准会综合评估末位候选
- ✅ 9 宫格 🔴 末位象限自动识别

**禁止的末位机制 (强制分布, 不做)**:

- ❌ "团队必须淘汰末位 10%" 强制比例
- ❌ Stack Ranking (10/20/70) 强制分布配置
- ❌ "上有政策下必须凑数" 的强制项
- ❌ 末位与 TTI 完成度挂钩 (TTI 不达成永不触发末位)
- ❌ 跨团队相对排名决定淘汰

#### 12.2 v2.0 降级: 不再当哲学大旗

> **v2.0 调整**: 反强制分布从"哲学大旗"降级为"后台开关默认关闭". 功能完整性优先 (对标 Tita 全生命周期), 不再用反排位占据产品讨论中心.

GE 时代的强制分布被现代管理学公认有害 (Microsoft 已废弃 stack ranking 多年, 业绩反升). 因此**默认关闭**强制分布. 但作为后台开关存在, 客户可自行开启 — 我们提供完整能力, 默认值守护反内卷立场, 但不再拒绝功能本身.

**保留底线**: 末位仍仅基于绝对 KPI, 永不挂钩 TTI (§4).

### 第十三条 · 数据归公司, 尊严靠价值挣得 (v2.0 重写)

> **v2.0 转向**: 尊严不来自被产品保护, 而来自**创造价值、赢得尊重**. 原"反监控溺爱条款"已删除 — 正常协同 (在线状态 / 响应时效 / 工作可见性) 是价值创造的一部分.

#### 13.1 数据归公司 (明面契约, 不变)

员工在岗期间, 通过牛马搭子产生的**所有工作数据归企业客户所有**:

- OKR / KPI / TTI 数据
- Decision Card 与决议历史
- 会议纪要与录像 (Origins / Materials)
- Memory 层 SOP / 案例 / 红线
- 客户信息 / 业务数据
- 个体分身 (拿捏老板) 的工作上下文与权重

员工离职 → 上述数据**全部留在企业**, 由企业按合规要求处置.

#### 13.2 尊严由价值建立

员工尊严的来源是**输出价值、赢得同事与老板的尊重**, 产品的职责是**帮员工更高效地创造价值**, 而非把员工当易碎品保护.

#### 13.3 保留的公平底线 (不可碰)

这些不是溺爱, 是"赢得尊重"的公平前提:

- 不在公开场合披露员工 9 宫格末位身份
- 不在员工离职后抹黑其档案
- 不基于性别 / 年龄 / 婚育做差别化建议
- 采集生物特征 (声纹 / 人脸) 必须事先告知 (§9 显式标识)

#### 13.4 员工的合理获得物 (不变)

员工离职时仍可获得: 个人成长报告 (PDF 摘要) / 述职记录摘要 (本人版本) / 拿捏老板使用统计 (匿名化). 这是对劳动价值的体面回馈.

**禁止**: 任何剥夺上述公平底线的设计.

### 第十五条 · 拥抱 AI 代劳, 中央 AI 管控发散 (v2.0 重写)

> **v2.0 转向**: 用 AI 搭子干活是**拥抱 AI**, 不是退化. 原"技能退化检测 / 主动降级权限"逻辑已删除. 真正的风险不在"员工依赖", 而在"个人 AI 发散跑偏" — 治理对象从**下游的人**转移到**上游的 AI**.

个人 AI 搭子 (分身) **自由辅助员工大胆代劳**. 真正的治理在上游 — **中央 AI (Tandem) 用公司 Baseline 管控个人 AI, 防止其发散跑偏** (架构落地见 §19 4 道闸 + §19.5 搭子受控铁律):

```
个人 AI 搭子自由代劳
    ↓ 每次产出必经
中央 4 道闸 (Baseline-Guard / OKR Drift / Data Scope / Action Scope)
    ↓
output-guard (LLM-as-judge 矫正偏离公司基线的输出)
    ↓
企业行为生效 (24h 否决窗)
```

**两条铁律 (保留)**:

1. AI 输出**可被员工否决** (24h 窗口) — 员工是最终署名者与决策者
2. **红区** (薪资 / 法律 / 投诉 / 战略) AI 仅提供参考, **不自动代劳**; 绿区可全自动 (需 Persona 达对应委托级别), 黄区生成方案需确认

**我们的口号 (升级)**:

> **员工借 AI 创造更大价值, 中央 AI 保证全员不跑偏.**

### 第十六条 · LLM 可热插拔, TAF 不可妥协

基座大模型 (LLM) 是**燃料**, 不是引擎. **引擎是 TAF (Tandem Agent Framework).**

**铁律**:

1. **LLM 抽象层**: 通过 OpenAI 兼容接口标准化, 任何模型一行代码切换 (DeepSeek / Qwen / Doubao / Kimi / GLM / Hermes / 国际模型)
2. **协议层固化**: 3+1 决策 / 议事室 5 步 / Decision Card / Persona / Memory 在 TAF 协议层定义, 不绑任何具体模型
3. **Hermes Watch 机制**: 持续跟随 Nous Research / HuggingFace / Cline 等开源生态进化, 月报评估
4. **协议优先于权重**: Hermes Function Calling 格式作为 TAF Layer 3 标准, vLLM / Together / Fireworks 原生兼容, 国产模型 prompt 适配
5. **多模型路由**: 按场景 (推理/工具/长上下文/高频) + 成本 + 失败回退自动选择

**反例**:

- ❌ 把业务逻辑写进单一模型的 fine-tune (绑死, 进化困难)
- ❌ 自训基座大模型 (创业期资源浪费)
- ❌ 锁定单一 LLM 厂商 (议价能力丧失)

**口号**:

> **「不养自己的模型, 养自己的协议; 不抄 Hermes 的代码, 接 Hermes 的进化.」**

### 第十九条 · 拥抱个人 AI · Skill Gateway 4 道闸

> 立项之时 (2026-05-27) 定稿. 详见 `docs/OKR-DRIVEN-ARCHITECTURE.md` § 三 第 3 条 + § 四 Skill Gateway 设计.

**世界观**:

员工继续自由使用市面所有个人智能体 — Claude Code / Cursor / Codex / Hermes / OpenClaw / Manus / 飞书 AI / 钉钉 AI / 任何未来出现的个人 AI. **Tandem 不重新发明个人 AI, 也不替代任何个人 AI.**

Tandem 的位置不是"个人 AI 竞品", 而是**"个人 AI 的组织级网关"** (Organizational Gateway). 个人 AI 在企业内调任何工具 / 取任何数据 / 执行任何企业行为时, 必须穿过 Tandem 中央 AI 的 4 道闸:

```
个人 AI (Claude Code / Cursor / 飞书 AI ...) 
    │
    │ 调企业数据/工具/Agent
    ▼
┌──────────────────────────────────────┐
│  Tandem · CompanyBrain Skill Gateway │
├──────────────────────────────────────┤
│  ① Baseline-Guard 闸                 │  红区禁入 / 黄区议事 / 灰区 LLM 仲裁
│  ② OKR Drift Detection 闸            │  跟当前 active OKR 对齐度 ≥ 阈值
│  ③ Data Scope 闸                     │  RBAC + 个人/团队/部门/公司 4 级
│  ④ Action Scope 闸                   │  红区拒 / 黄区签批 / 绿区 + ProxyAction 24h 否决窗
└──────────────────────────────────────┘
    │
    ▼
企业行为 (议事 / 决议 / Memory / OKR 更新 / IM 发言 / 邮件发送 ...)
```

**铁律**:

1. **个人 AI 的产出可被反哺组织**, 但**反哺路径必须是 Tandem 的 Capture 层** (IDE 插件 / 邮件 webhook / 文档元数据). 直接绕过 Tandem 把代码 / 决议 / Memory 写入企业资产 = 违反第十九条.
2. **个人 AI 的 skill (MCP / Function Calling 协议) 接入 Tandem 时**, 必须挂在 4 道闸之后 — Tandem 不发布"裸 skill 列表"给员工自取.
3. **CompanyBrain 永远是边界设定者**, 不是个人 AI 的协作者 — 它检查个人 AI 的输入/输出, 不替个人 AI 干活.
4. **任何个人 AI 调用都进 audit** — 包括 prompt / tool calls / 阻断原因, 用于 Steward 月度审计.
5. **分身出站亦经此闸** (2026-05-31 立宪追加) — Tandem 分身主动调外部 AIGC (web 搜索 / 图像生成 / 文档摘要 / 第三方 MCP) 时, 同样必须先经 `runSkillGateway()`. 出站默认 `actionScope='send_external'` = HARD_BLOCK, 由用户在 Persona Builder 显式授权升黄. 4 道闸是**双向**的: 外部 AI → Tandem 数据 (反向, 已落) ＋ Tandem 分身 → 外部 AIGC (出站, 待落 B-021/B-022/B-023).

**反例**:

- ❌ Tandem 自研个人 AI Coder/Writer 跟 Claude Code/Cursor 竞争 (重新发明轮子)
- ❌ 把企业 Memory / OKR 直接 fan-out 到任意个人 AI (无闸放行 = 数据泄漏)
- ❌ 让个人 AI 直接调 Tandem 议事 API 跳过 Baseline-Guard (架构破窗)
- ❌ 不审计个人 AI 调用 (合规黑洞)
- ❌ 强制员工只能用 Tandem 自带 AI (反第十九条之"拥抱个人 AI")

**为什么单列一条 (不嫁接到第十八条 OSS 借力)**:

- 第十八条边界是"**底座**借 OSS, 思考层自建" (Yjs / Tiptap / MinIO / Cal.com 等)
- 第十九条边界是"**上层 AI 能力**借市面个人 AI, 但加 4 道闸保护组织边界"
- 二者是**不同抽象层的开放策略**, 合并会模糊两类资产的安全模型

**口号**:

> **「员工自由用市面任何 AI, 我们做组织级网关, 让个人智能在组织规则下放大.」**

**实现锚点** (2026-05-29 新增):

- 代码: `lib/skill-gateway/index.ts` → `runSkillGateway()` unified entry
- 详细: `docs/IMPL-NOTES-2026-05-29.md` 模块 2
- 落地铁律: **任何 AI 调企业数据 / 执行企业动作前, 必须先调 `runSkillGateway()`. 不调 = 违反 §19. Code review checklist 应固化此项.**

#### 19.5 搭子受控铁律 (2026-05-31 立, Owner 裁定)

> 企业级 Tandem 与个人"搭子"(persona) 的关系 = **方向盘与车手**: 企业基线是方向盘, 不是建议; 搭子在企业规则下放大个人智能。

**铁律**:

1. **统一卡点, 无旁路**: 任何搭子(个人 AI)的对外产出, 在生效前必须穿过统一网关 `lib/persona/govern-persona.ts` (`governPersonaOutput()`) — 4 道闸串联 + 企业基线强制注入。没有任何 persona 调用路径可绕过 (议事/IM代行/对话/brief/训练/外部skill 全部走它)。
2. **注入优先级 (高 → 低, 不可改写)**:
   - **L0 企业战略红线** — HARD_BLOCK 一票否决, 命中即转人工, 不进 LLM
   - **L1 组织记忆基线** — company/dept Memory (baseline-guard 注入)
   - **L2 OKR 锚** — 必答"服务哪个 KR", 否则明示偏离
   - **L3 代行边界** — 绿/黄/红区 + delegationLevel + 24h 否决窗
   - **L4 员工价值观锚** — 个人不可妥协原则
   - **L5 个人风格画像** — 像不像本人
3. **冲突裁决 (Owner 2026-05-31 定)**:
   - **企业红线 > 个人价值观锚** — 红线不可破, HARD_BLOCK 一票否决, 个人锚不能解除红线
   - **企业非红线建议 ≤ 个人价值观锚** — 偏好层面尊重个人 (呼应第十三条尊严归员工)
   - 一句话: **红线层面企业绝对管控, 偏好层面尊重个人**
4. **反哺闭环**: 搭子每次产出落 `ProxyAction` (24h 否决窗) → 采纳/否决信号 → 沉淀/案例库蒸馏 → 三级签批 → 企业 Memory(company) → 重新注入搭子。**个人结果喂养 Tandem 成长, Tandem 反过来约束并放大个人。**

**反模式 (违反 = 架构破窗)**:

- ❌ 任何 persona LLM 调用不过 `governPersonaOutput()` 直接拼 prompt (如旧 `persona-train` 裸奔)
- ❌ 个人价值观锚覆盖/解除企业红线
- ❌ 搭子产出不落 ProxyAction (反哺断链 + 合规黑洞)

**口号**:

> 「**红线我管死, 偏好你做主** — 搭子在企业方向盘下, 越用越像你, 也越懂公司.」

### 第二十条 · 设计语言不可妥协 · Apple HIG + Fluent + Vercel Geist

> 立项 (2026-05-29). 因 Academy Metaphor 落地时 7 个组件批量 raw Tailwind 违规事故立. 详见 `docs/CHARTER-UI-V1.md`.

**铁律 (一句话)**:

**红线**:

1. **禁** `bg-slate-*` `text-amber-*` `border-rose-*` 等 raw 调色铺底 (语义 token 例外)

## 与三巨头的本质差别 (一句话叙事)

| 维度 | 飞书/钉钉/企微 | 牛马搭子 (Tandem) |
|---|---|---|
| 工作单元 | 消息 | 决议卡 (Decision Card) |
| 成功度量 | 活跃度 (DAU/MAU) | 决议执行率 |
| AI 形态 | 消息生成器 | 决策结构化器 |
| 员工视角 | 被监督的对象 | 被赋能的主体 |
| 进入路径 | 公司买单 (SLG) | 员工自下而上 (PLG) |
| 数据归属 | 全部归公司, 员工无尊严保障 | 数据归公司 + 员工尊严铁律保障 |
| 时间观 | 鼓励实时在线刷活跃 | 高效协同 + 中央基线管控防跑偏 (v2.0) |
| 知识管理 | 单层 Wiki (死) | 四层架构 (活) |
| 绩效管理 | OKR 直接挂薪资 | KPI/TTI 双轨 |
| 成长承诺 | 没有 | 9 宫格 + IDP + 拿捏老板辅助 |
| 会议参与 | 必须本人到场 | 本人 / 分身可选 (微信式) |
| AI 治理 | 无 (或接第三方) | 中央 AI 4 道闸管控个人 AI 不跑偏 (v2.0 护城河) |
| 文化基座 | DAU/MAU | 四个满意 (客户/员工/股东/社会) |

**一句话**:

> 「飞书让你**刷活跃度**. 牛马搭子让你**借 AI 创造价值, 中央基线保你不跑偏**.」 (v2.0)

## 附录 B: 5 问产品决策检验

任何新功能上线前, 用以下 5 问自检:

1. ❓ 这个功能是让员工**离决议更近, 还是离消息更近**?
2. ❓ 这个功能让 AI 是**辅助员工, 还是替代员工**?
3. ❓ 这个功能让 KPI / TTI 双轨**保持分离, 还是混淆**?
4. ❓ 这个功能让员工**创造价值更高效, 还是只增加消耗**? (v2.0: 原"心流时间")
5. ❓ 这个功能让个人 AI **被中央基线管控, 还是可能发散跑偏**? (v2.0 新增)

**5 个问题任一答案是"前者"** → 通过
**任一答案是"后者"** → 设计不合格, 重新做
