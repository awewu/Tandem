# Tandem 自建飞书级底座 · 技术路线与执行计划

> **版本**: v1.0 (2026-05-17)
> **前提**: 不寄生飞书, 不自建飞书轮子, 用开源底座 + 自建差异化 = 飞书级功能架构
> **性质**: 产品战略重定位文档 — 从"寄生"升级为"自建对标"

---

## 一、战略重定位

### 1.1 用户原话

> "我要的并不是和飞书接通, 而是要**具备飞书的功能架构体系**.
> 再加上我们**独有的聚焦企业目标达成的驱动架构**."

### 1.2 这意味着什么

```
飞书 = 底座层 (IM+文档+日历+会议+云盘+搜索+工作台+移动端)
     + 应用层 (审批/考勤/OKR 轻量/文档)
     + 生态层 (钉企飞上架/小程序/应用市场)

Tandem = 底座层 (用开源快速补齐 70-80%)
       + 驱动层 (自建 —— 这是飞书没有的)
       + 应用层 (自建 —— 这是飞书做不好的)

驱动层: 议事室 / 3+1 决策 / Memory 签批 / Persona 双层 / 拿捏老板
应用层: 重型 OKR / 9 宫格 / 日报闭环 / 四层知识 / TAF Agent
```

### 1.3 与 MANIFESTO §18 的兼容性

**MANIFESTO §18 原文**:
- 自建 (核心差异化): OKR/TTI/DC/9宫格/议事室/3+1/拿捏/TAF/知识/Steward/Inbox
- 自建 (复用 Hermes): IM/群聊/组织架构/邮件
- 寄生 (调 API): 音视频会议 (腾讯会议)
- 永不做: CRM/OA/审批/考勤/外勤/GPS/印章

**用户新要求** 与 §18 **不冲突**, 但扩展了"自建"的边界:
- IM: 从"自建 (复用 Hermes)" 升级为 **全功能 IM** (对标飞书)
- 文档: 从"未提及" 升级为 **实时协同文档** (对标飞书文档)
- 日历: 从"Cal.com fork" 升级为 **日历+会议室** (对标飞书日历)
- 云盘: 从"MinIO+自建UI" 升级为 **文件管理** (对标飞书云盘)
- 搜索: 从"未提及" 升级为 **全局搜索** (对标飞书搜索)

**唯一矛盾**: 音视频会议.
- §18 写"寄生 (调 API): 腾讯会议"
- 用户要求"具备飞书级功能架构"
- 飞书的音视频会议是自研的 (非调用 API)

**建议**: 短期 (V1) 保留腾讯会议 API, 中期 (V2) 评估自建 WebRTC 或集成 Jitsi.

## 二、飞书功能架构拆解

### 2.1 飞书 11 大功能域

| 域 | 飞书实现 | Tandem 目标 | 技术路线 |
|---|---|---|---|
| **IM** | 自研, 富文本/撤回/编辑/引用/转发/搜索/已读 | 全功能 IM | **自建** (基于当前 InMemory → PG + SSE → V2 上 Redis) |
| **群聊** | 自研, 多类型/成员管理/公告/文件/日历 | 全功能群聊 | **自建** |
| **文档 (Doc)** | 自研, 实时协同/评论/版本/模板 | 实时协同文档 | **Yjs + Tiptap** (MIT, 成熟) |
| **多维表格** | 自研, 类 Airtable | V2 再评估 | **Univer** (Apache 2, 中国开源 Excel) |
| **日历** | 自研, 日程/会议室/共享/邀请/重复 | 日程管理 | **Cal.com** (AGPL, 需法务) 或 **自建** |
| **音视频会议** | 自研, 高清/录制/转写/字幕/白板 | 一键开会 | **腾讯会议 API** (V1) → **Jitsi** (V2) |
| **云盘** | 自研, 文件夹/权限/预览/版本/分享 | 文件管理 | **MinIO + 自建 UI** |
| **通讯录** | 自研, org chart/批量/权限/搜索 | 组织架构 | **自建** (已有基础) |
| **搜索** | 自研, 全局/消息/文档/人/应用 | 全局搜索 | **PG 全文检索 + pgvector** |
| **工作台** | 自研, 应用市场/机器人/SSO/API | 应用集成 | **Launchpad + 开放 API** |
| **移动端** | 自研 iOS/Android | 移动端 | **React Native** (V2) |

### 2.2 关键决策

**决策 1: IM 全自建 vs fork Rocket.Chat**

| 方案 | 优势 | 劣势 | 推荐 |
|---|---|---|---|
| **A: 继续自建** | 与 Tandem 决议驱动深度集成; 无外部依赖 | 工作量大 (~3-4 周补全基础功能) | ✅ **采纳** |
| B: fork Rocket.Chat | 功能现成; 10 年成熟 | Meteor 框架老; 与 Next.js 异构; 改造成本高 | 已废弃 (2026-05-08) |
| C: fork Mattermost | Go 性能好; 功能全 | 商业版限制; Go + React 异构 | 备选 |

**决策**: 继续自建 IM. 当前已有频道/消息/SSE/mention/撤回/已读骨架, 补全编辑/引用/转发/搜索/免打扰即可.

**决策 2: 文档协同**

| 方案 | 优势 | 劣势 | 推荐 |
|---|---|---|---|
| **A: Yjs + Tiptap** | 实时协同成熟; MIT 协议; React 生态 | 需自建 UI 壳; 表格能力弱 | ✅ **V1 Doc** |
| B: Univer | 中国开源 Excel; Apache 2; 表格强 | 实时协同不如 Yjs 成熟 | ✅ **V1 Sheet** |
| C: 自研编辑器 | 完全可控 | 工作量巨大, 不现实 | ❌ |

**决策**: Yjs + Tiptap 做 Doc (V1), Univer 做 Sheet (V1.5).

**决策 3: 音视频会议**

| 方案 | 优势 | 劣势 | 推荐 |
|---|---|---|---|
| **A: 腾讯会议 API (V1)** | 成熟稳定; 中国本土; 分身代参可集成 | 非自建; 依赖腾讯; 费用 | ✅ **V1** |
| B: Jitsi Meet | 开源; 可私有化; 功能全 | 部署复杂; 中国网络体验差 | V2 评估 |
| C: 自研 WebRTC | 完全可控 | 音视频是地狱级难度 | ❌ |

**决策**: V1 保留腾讯会议 API, V2 评估 Jitsi.

## 五、诚实的工作量评估

### 5.1 与飞书的差距不是"不能做", 是"需要时间和人"

| 维度 | 飞书投入 | Tandem 预计投入 |
|---|---|---|
| 团队规模 | 数百人 × 5 年 | **4-6 人 × 6 个月** |
| 总工时 | > 1000 人月 | **~ 60 人月** |
| 技术策略 | 全自研 | **开源借力 + AI 加速** |
| 质量目标 | 极致体验 | **够用即可, 差异化取胜** |

### 5.2 风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| Yjs 实时协同稳定性 | 中 | 高 | 早期用 y-indexeddb (本地), 后期上 y-websocket |
| Cal.com AGPL 法务 | 中 | 高 | V1.5 自建日历替换 |
| 腾讯会议 API 限制 | 低 | 中 | V2 评估 Jitsi |
| React Native 开发周期 | 高 | 中 | Phase 6 单独排期, 不影响 V1 GA |
| 等保测评周期 | 高 | 中 | 提前启动, 外包 + 并行 |
| 资金不足 | 中 | **致命** | 6 个月 300-400 万, 需确认 |

### 5.3 关键假设

1. **AI 加速 3x**: Cascade/Cursor 加持下, 实际产能 3x 传统团队
2. **开源底座成熟**: Yjs/Tiptap/Cal.com/MinIO 足够稳定
3. **不追求完美**: 底座"够用"即可, 资源聚焦差异化
4. **资金到位**: 6 个月 300-400 万 RMB

## 七、结论

**用户要的不是"寄生飞书", 而是"自建一套飞书级底座 + Tandem 独有驱动层".**

这在技术上是**可行的**, 在战略上是**正确的**:
- 飞书底座是"必要非充分" — 没有底座, 差异化无法落地
- Tandem 差异化是"充分条件" — 有底座 + 差异化 = 不可替代
- 6 个月 4-6 人 300-400 万 = 现实可达

**关键成功因素**:
1. 资源聚焦: 底座"够用"即可, 不追求完美
2. 开源借力: Yjs/Tiptap/MinIO/Cal.com 省 70% 时间
3. AI 加速: Cascade/Cursor 省 50% 编码时间
4. 顺序正确: 先 PG 持久化, 再 IM 补全, 再文档, 再日历, 最后移动端
5. 资金到位: 6 个月不能断粮
