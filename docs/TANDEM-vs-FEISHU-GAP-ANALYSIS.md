# Tandem vs 飞书 · 协同功能差距分析

> **版本**: v1.0 (2026-05-17)
> **前提**: 飞书是"消息驱动"协同的成熟标杆, Tandem 是"决议驱动"协同的实验品.
> **本分析不评判优劣, 只列差距.**

---

## 一、核心结论（TL;DR）

| 维度 | 飞书 | Tandem |
|---|---|---|
| **哲学** | 消息驱动 (Message-First) | 决议驱动 (Decision-First) |
| **最小单元** | 消息 | Decision Card |
| **成功度量** | DAU/MAU / 消息量 | 决议执行率 / 17min 成交率 |
| **成熟度** | 生产级 SaaS, 数百万企业客户 | PoC 原型, InMemory, 未商用 |
| **功能完整度** | ~95% (企业协同全场景覆盖) | ~40% (核心独创模块跑通, 底座薄弱) |

**关键洞察**: Tandem 与飞书不是"功能差距", 而是"范式不同". 飞书把"沟通"做到极致, Tandem 试图把"决议"做到极致. 但 Tandem 的"沟通底座"远未达到可用水平.

## 二、功能矩阵 · 逐项对比

## 五、商业落地建议

### 5.1 诚实面对差距

**对客户说**: "Tandem 不是飞书的替代品, 而是飞书做不到的事."

| 场景 | 飞书 | Tandem |
|---|---|---|
| 日常聊天/发文件/开会 | ✅ 极好 | ⚠️ 勉强可用 |
| OKR 管理 | ⚠️ 基础 | ✅ 重型 5 层 + AI 纠偏 |
| 决议会议 (议事室) | ❌ 无 | ✅ 17min 硬上限 + 3+1 |
| 知识管理 (SOP/案例) | ⚠️ Wiki | ✅ 4 层 + 签批 + 降级 |
| 员工成长 (TTI/Persona) | ❌ 无 | ✅ 双层 + 5 阶段 |
| 人才盘点 (9 宫格) | ❌ 无 | ✅ KPI × TTI 矩阵 |
| 保护员工反卷 | ❌ 无 | ✅ 宪章 18 条 |

### 5.2 落地策略：寄生而非替代

**不要试图"替代飞书"** — 这在 3 年内不可能.
**应该"寄生在飞书上"** — Tandem 做飞书做不到的事.

```
企业已有飞书 ──► 保留飞书用于 IM/文档/会议/日历
              │
              ▼
         Tandem 接入
              │
              ├─ 飞书 OAuth 登录 (SSO)
              ├─ 飞书机器人推送 "Waiting" 状态 (EVO-19)
              ├─ 飞书日历同步 (会议预约)
              ├─ 飞书文档嵌入 (iframe 或链接)
              │
              ▼
         Tandem 独有功能
              │
              ├─ 议事室 + 3+1 决策
              ├─ OKR 重型管理
              ├─ Memory 签批流
              ├─ Persona 双层
              ├─ 9 宫格人才矩阵
              └─ 拿捏老板
```

**这样企业的迁移成本 = 0**, Tandem 只是飞书工作台里的一个应用.

## 六、结论

### Tandem 与飞书的差距不是"功能多少", 而是"范式不同 + 底座薄弱".

**飞书是"沟通工具做到极致"**.
**Tandem 试图"决议工具做到极致", 但沟通底座远未达标.**

**诚实评估**:
- 如果客户问"能替代飞书吗" → **"不能, 也不想替代"**
- 如果客户问"飞书缺什么你补什么" → **"OKR 重型管理 / 议事室决策 / 知识签批 / 员工 AI 成长"**
- 如果客户问"需要多久能用" → **"寄生飞书: 1 个月; 独立替代: 12 个月"**

**最终建议**:
1. **放弃"替代飞书"的幻想**, 拥抱"寄生共生"策略
2. **优先补齐 P0 底座** (DB/搜索/富文本/文件), 让产品"可用"
3. **通过飞书机器人/Webhook 做 Gateway** (EVO-19), 降低用户切换成本
4. **把独创模块做深做透** (议事室/OKR/Memory/Persona), 建立不可替代性
