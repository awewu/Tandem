# 后端数据库专项审计报告

- **项目**: Tandem（牛马搭子）
- **日期**: 2026-06-09
- **范围**: `lib/storage`, `lib/infra`, `lib/repositories`, `drizzle/migrations`, 数据访问层调用方
- **状态**: 已审计 + 已落地优化（见文末"修复落地记录"）

---

## 一、架构总览

数据层为**双轨制**：

- **轨道 A — `KvStore` 通用 JSONB 表** (`lib/storage/drizzle-store.ts` `DrizzleKvRepository<T>`):
  绝大多数模块（Persona / Memory / OKR / IM / 1on1 / 360 / 治理 / 手抄… 约 40+ 集合）都落在
  一张 `(collection, id, data jsonb)` 表里。
- **轨道 B — 强类型表**: KPI 8 张、Academy 8 张、Document/Calendar/Drive/Notification、User、
  AuditLog、UsageEvent、LlmUsageLog。规范列 + 索引 + 软删齐备。

连接层: `postgres-js` + Drizzle，连接池 `max: 10`，`prepare: false`
(`lib/infra/drizzle-client.ts`)。启动有生产硬化 guard，DB 初始化失败在 production 下
**fail-loud** (`lib/boot.ts`)。

**结论先行**: 轨道 B 工程质量高；轨道 A 是风险集中区，尤其在租户隔离与规模化性能两个维度。

## 二、发现（按严重程度分级）

### C1 — Critical: KvStore 租户隔离在数据层完全缺失

`DrizzleKvRepository.create()` 从不写入 `tenantId` 列 → 每行 `tenantId` 永远是 `'default'`，
表上的 `tenantId` 列 + `KvStore_tenant_idx` 索引是死的。`list(filter)` 只按 `collection` 查、
把整张集合拉进内存再 JS 比对；`get/update/delete` 只按 `(collection, id)` 定位，无租户维度
→ 拿到任意租户合法 id 即可跨租户读/改/删。

### C2 — High: `list()` 全集合扫描 + 内存过滤，无 LIMIT、无分页

每次列表请求 `SELECT * FROM KvStore WHERE collection=?`，无字段下推、无 LIMIT。增长型集合
(im_messages / memories / check_ins / audit) 随数据量线性膨胀内存与网络。强类型 KPI 表 `list()`
同样"先全表 SELECT 再 JS filter"。

### C3 — High: 按 id 的读改删无租户校验，靠路由各自自觉

跨租户防护被推到每个路由处理器。有的补了 (`360/cycles/[id]` 校验 `cycle.tenantId`)，但属
分散、靠纪律的防线，漏一个即越权。数据层无兜底。

### C4 — High: 热路径 JSONB 查询无表达式索引

`findByRefreshHash` (每次刷 token) / `findByHash` (邀请) 走 `data->>'field'` 顺序扫描，
对整个集合 seq scan，无 GIN/表达式索引。

### C5 — Medium: User 读取不过滤 `deletedAt`（软删失效）

`findByEmail/findById/list` 无 `deletedAt IS NULL`。登录链路只查 `user.disabled`。若用户被软删
但 `disabled` 未同时置真，仍能被命中甚至登录 → 潜在认证绕过。

### C6 — Medium: KPI 强类型表 list 不按 tenantId 过滤

KPI 表有 `tenantId` 列但 `list()` 只下推 `cycleId`/`assigneeId`，租户靠 JS 兜。传外租户
`cycleId` 即可跨租户读。

### C7 — Low / 观察项

- `_storeKind: 'prisma'` 历史误命名（实为 Drizzle），与 boot 比较逻辑耦合，易误导。
- `prepare: false` 关闭预编译，牺牲计划缓存。
- `generateId` 用 `Math.random()`，仅可用于实体 id，不可作安全令牌。
- `auth_event` 1% 概率分摊剪裁的 `catch` 块为空，静默吞错。

### 做得好的地方（明确肯定）

- **SQL 注入: 安全**。全部 `sql\`...\`` 走 Drizzle 参数化绑定，唯一裸 SQL 只含字面常量。
- 强类型表索引完善（AuditLog/UsageEvent/LlmUsageLog/Course 有复合 `(tenantId, …)` 索引）。
- 审计日志哈希链 + 每租户单调 seq，满足等保/PIPL 证据要求。
- 生产启动 fail-loud，不会静默退回内存存储丢数据。

## 三、维度评分

| 维度 | 评级 | 说明 |
|---|---|---|
| SQL 注入防护 | 良好 | 全参数化 |
| 多租户隔离 | 薄弱 | KvStore 数据层零隔离，靠路由自觉 |
| 查询性能/可扩展 | 偏弱 | 全集合扫描 + 内存过滤，热路径无索引 |
| 软删一致性 | 中 | User 读不过滤 deletedAt |
| 强类型表设计 | 良好 | 列/索引/软删/审计齐备 |
| 启动健壮性 | 良好 | production fail-loud |

## 四、修复优先级建议

1. **P0** — 给 `DrizzleKvRepository` 注入租户维度（create 落列 + list/SQL 下推），数据层兜底。
2. **P1** — `list()` 加分页/LIMIT，高频集合过滤下推 SQL。
3. **P1** — 为 `auth_session`/`auth_invite` 的 JSONB 查询建表达式索引。
4. **P2** — User 读取统一加 `deletedAt IS NULL`，确认删除流程同时置 `disabled`。
5. **P2** — KPI/Academy `list()` 把 `tenantId` 下推到 WHERE。
6. **P3** — 清理 `_storeKind` 命名、补 catch 日志、评估 prepare 策略。

## 五、修复落地记录（2026-06-09）

- **C1/C3 根因**: `DrizzleKvRepository.create()/update()` 现从 `data.tenantId` 落 `tenantId` 列；
  迁移 `0006` 回填存量行 `tenantId = COALESCE(data->>'tenantId','default')`，使列与索引"复活"。
- **C2/C1 list 下推**: `DrizzleKvRepository.list()` 在 `filter.tenantId` 存在时把
  `eq(KvStore.tenantId, …)` 下推到 SQL，减少加载行数（兜底 + 性能）。
- **C4**: 迁移 `0006` 新增 `auth_session.refreshTokenHash` / `auth_invite.codeHash` /
  `auth_event.userId` 的 JSONB 表达式 partial 索引。
- **C5**: User `findByEmail/findById/list` 增加 `deletedAt IS NULL`。
- **C6**: KPI `list()` 在 `filter.tenantId` 存在时下推到 SQL。
- **C7**: 空 `catch` 补告警日志。
- **P1 分页 (opt-in)**: `Repository.list` 增加可选第二参 `opts?: ListOptions { limit?, offset? }`，
  向后兼容（既有 `list(filter)` 调用与实现均不受影响）。`DrizzleKvRepository.list` 在"仅租户过滤"
  时把 `limit/offset` 下推到 SQL，否则退化为 JS 过滤后切片；`InMemoryRepository.list` 同步支持。
  热集合 (im_messages / memories / audit) 后续可逐步传 `limit` 以避免全集合加载。

> 注: KvStore `get/update/delete(id)` 仍以 `(collection,id)` 为主键定位；id 为
> collision-resistant 随机串，跨租户按 id 直取风险低，路由层已有 `tenantId` 复核。
> 全面把租户维度并入 by-id 接口签名属更大重构，列为后续。
