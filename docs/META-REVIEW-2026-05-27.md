### 真实问题 2: 你担心 Hermes 底座的"血统"

征兆:
- "我记得我是按照 Hermes 的代码底座给你做的驱动引擎, 目前有没有变化?"
- "后续 Hermes 的相关开源升级, 你如何同步?"

→ 这个担心实际**没有事实依据**——git log 显示这是从 0 起步的项目, 不是 Hermes fork。但你担心的真问题是"未来 Hermes 出新版本, Tandem 会不会被甩开 / 被卷死"。

**真实答案**: Tandem 跟 Hermes 是 API consumer/provider 关系, 不是 fork。Hermes 升级 = 升级 hermes 二进制, Tandem 不动。但要把 `lib/hermes-api.ts` 包装成纯 contract + adapter (这是 B-007), 才能真做到"Hermes 演进与你无关"。

### 真实问题 3: 你担心被定位成"飞书的影子"

征兆:
- "你保留了飞书等接口, 他们对软件的价值是什么?"
- "我们说要替代, 做不到吗?"

→ 你担心的真问题是: **市场会不会把 Tandem 误读为"AI 化的飞书"**, 从而把我们当作小厂被巨头碾压。

**真实答案**:
- 代码层面没有"依赖飞书"——`lib/tita-adapter.ts` 是数据迁移层, OSS 借力是替代飞书通用模块, 我们的差异化模块 (Memory / Persona / 议事 / OKR-TTI) 100% 自建
- 但**外部观感是真实风险**——投资人 / 客户第一眼看 UI 像飞书。这个不是改代码能解决的, 是产品定位 + 市场叙事问题
- 解法: 对外讲故事时**不讲"替代飞书"**, 讲"AI-Native 协作 + 让你从飞书迁移过来"。前者会被技术人看穿, 后者更准确

### 真实问题 4: 你想知道核心创新有没有被"保住"

征兆:
- "企业级记忆驱动个人 AI 分身是我们做企业级 AI 的核心功能。你有没有保障好?"

→ 这是 Owner 最深的担忧——**怕辛辛苦苦写的核心被自己 / 协作者 / 时间稀释了**。

**真实答案**:
- 代码事实层面: Memory + Persona 共 20+ 文件, 是被认真对待的核心 (REFLECTION §三)
- 创新事实层面: Memory 4 层升降级 + Persona Evolution 业内没第二家在做
- 但**保障度有缺口**——缺 Persona 评测体系 / 缺成本观测 / 缺用户行为数据。**核心是"在", 但"是否在生长"无法证明**。这是下季度第一优先级该补的

### 真实问题 5: 你担心被 Claude Memory 等大厂能力卷死

征兆:
- "Claude 新发的关于记忆的新能力, 你有没有学习进化?"

→ Owner 看到大厂出新能力, 自然焦虑——"我做的这套是不是已经过时了?"

**真实答案** (REFLECTION §七 详细论证):
- Claude Memory 是 consumer 单用户范式, 跟 Tandem 企业多人 4 层流转是**不同物种**

## 五、对话中暴露的隐含矛盾 / 不一致

诚实审视, 今天对话里有 3 处自相矛盾的地方:

### 矛盾 1: "替代飞书" vs "OSS 借力"

**矛盾**:
- ROADMAP.md / OKR-VS-TITA.md / WECOM-FEATURE-MAPPING.md 等多处暗示"替代飞书企微 Tita"
- 但实际架构是"OSS 借力（用 Cal.com / Etherpad / MinIO 等开源组件填飞书的非差异化部分）"
- **这两件事在本质上不同**: 借力 ≠ 替代

**该如何处理**:
- 内部认知: 我们替代的是**飞书的核心场景** (议事 / OKR-AI / Memory / Persona), 不替代它的基础设施 (IM / 日历 / 文档底层)
- 对外叙事: 不说"替代", 说"AI-Native 协作平台 + 数据可从飞书迁移"
- → 应该把这条写成 ADR: `docs/adr/0001-substitution-vs-borrow.md`

### 矛盾 2: "PWA 已就绪" vs "Tailscale 是 HTTP"

**矛盾**:
- 之前 Phase 4/5 完成时说"PWA 已就绪, 可装到手机主屏当 App 用"
- 但今天选的 Tailscale 路径是 HTTP, 部分 PWA 能力 (推送 / install prompt) 受限

**该如何处理**:
- "PWA 已就绪" = manifest/sw/icons 三件套对了
- 但**完整 PWA 体验** = HTTPS 部署。Tailscale HTTP 只是开发/试用, 不是产品形态
- → 应该明确"试用阶段用 Tailscale" + "产品阶段必须 HTTPS"。这条进 LAUNCH-CHECKLIST 的"上线前必查"

### 矛盾 3: "工程化基础是命门" vs "还没启动 CI"

**矛盾**:
- REFLECTION §四 我自己说"工程化和商业化是命门, 没 CI 是 Bus factor 风险"
- 但今天我们做完所有讨论, **CI 还是没建**

**该如何处理**:
- 这个矛盾**完全是因为时间分配**: 今天投入都在"想清楚"层面, 没投入"做掉"层面
- → 下次会话第一件事: 建 CI (`.github/workflows/test.yml`), 半天工作量。不要再让"想清楚"和"做掉"脱节
