## 一、什么是"完全超过 Cowork"(定调)

Cowork 的本质 = **个人主权的桌面执行体**: 本地跑、操作你的文件/应用、手机异步派活、
Projects 工作区、Plugins/Connectors、Sub-agents。它强在"替你把活干完"。

> ⚠️ "完全超过" **不是**"在桌面自动化上比腾讯/Anthropic 更能干" —— 那是拿我们的短板
> 拼对手的 infra 主场, 必输, 且会把自用阶段资源烧光。

**正确定义**: 搭子是一种更高级的工作伙伴物种 ——
**会成长 (5 阶段进化) + 被组织赋能 (能在公司系统里有效力地行动) + 会反思自纠 (真学习) +
全量工作上下文常驻受治理云端**。Cowork 再强也只是"聪明的临时工"; 搭子是"会进化的、
有组织授权的、越用越懂你的数字同事"。

## 三、搭子超过 Cowork 的三条命脉 (赢在它赢不了的地方)

### 命脉① 会进化 + 会反思自纠 (最高战略杠杆)

Cowork 有 Projects/memory, 但**不按用户做归因式学习**。搭子的 5 引擎飞轮
(能力/知识/风格/战略/反思) 才是物种级差异。其中**反思引擎 B-024 是根**:

> 假学习 = 计数器累加。真学习 = 归因诊断 (knowledge_gap / style_drift / skill_misuse)
> → 反推调整前 4 引擎 → 下次更准。

做出来后, 搭子"越用越像你、越用越准"; Cowork 永远"每次重新理解你"。

### 命脉② 在组织里行动, 不只动本地文件

Cowork 顶天整理你电脑的文件、生成 PPT。搭子能 (按阶段解锁) **开议事室、锚定 KR、
提交带 24h 否决的承诺、把产出沉淀进组织 Memory**。它的产出有**组织效力**, 不是孤岛文件。

### 命脉③ 全量工作上下文常驻受治理云端 (见 §五)

通过"工作盘同步盘", 员工每一份工作文档持续躺在云端大脑里 —— 搭子永远有全量上下文、
持续学习、CompanyBrain 持续治理。Cowork 单会话/本地/用完即走, 架构上补不了。

## 四、桌面端 = 工作盘同步盘 (OneDrive 式), 不是本地执行 agent

> Owner 2026-06-07 定调: "员工有账号 → 有桌面端, 但只对应自己的工作盘文档板块,
> 类似 OneDrive。个人资料同步云端, 中央大脑自动做上下同步。**主要工作在云端**。"

### 模型

- **桌面端 = 同步盘**: 登录后本地一个 OneDrive 式"工作盘", **只对应员工自己的工作文档板块**,
  不是任意本地文件系统访问。复用已有 Tauri 桌面壳。
- **主战场在云端**: 真正的 AI 工作 (搭子推理 / 议事 / 反思 / 治理) 跑云端, 桌面只是
  **采集 + 交付端面**。
- **中央大脑编排同步**: CompanyBrain 自动做"上下同步"(local↔cloud)。
- 正好补上 `CENTRAL-AI-ARCHITECTURE.md` 缺器官 **#17 (个人 AI 产出 Capture 层)**。

### 三项已拍板的设计决策 (Owner 2026-06-07)

| # | 决策 | 取值 | 落地含义 |
|---|---|---|---|
| ① | **工作盘归属** | **默认组织可见** (company-scope) | 与"数据归公司"一致。中央大脑可检索、可用作组织知识。**但"可见 ≠ 基线"**: 升级为治理基线 (能 HARD_BLOCK 的 procedural Memory) 仍走 promotion-flow 策展/签批, 防每份草稿变红线 |
| ② | **同步粒度** | **整盘镜像 + 后台按岗位配置** | 默认整盘同步; Steward/admin 后台按岗位工作性质配置哪些岗位同步、范围多大, 省公司存储 (有些岗位不需要浪费内存) |
| ③ | **桌面端能力** | **双向** | 上行: 工作文档进云端受治理大脑; 下行: 云端产出的成品写回本地交付目录 (对标 WorkBuddy"PC 端出成品", 但成品云端生成、只同步落地) |

### 红线处理 (与 `/atlas` G2 自洽)

G2 = 中央 AI 不读个人 IM/邮箱/通用 AI 沙盒。工作盘**默认 company-scope = 公司数据, 不在
G2 的"个人私域"桶内**, 因此自洽。要拆清两层动作:

- **同步编排 (搬字节)**: CompanyBrain 管"哪些目录、何时同步、冲突怎么解" —— 物流, OK。
- **基线化 (升治理规则)**: company-scope 文档可被中央大脑**检索/RAG**, 但升级为**基线
  procedural Memory** (能阻断他人) 仍需策展签批。

## 六、为什么"借 Cowork/WorkBuddy 当手脚"被否决

> Owner 2026-06-07 质询: "我们是竞争对手, 怎么借?" —— 这条质询直接修正了早期错误表述。

- **技术方向反了**: MCP 模型里 Host/Client = 大脑 (决定调什么工具), Server = 工具/手脚。
  Cowork/WorkBuddy/Claude Code 都是 **Host (大脑)**, 自己决策去调别人的 MCP server,
  **不是可被反向驱动的手脚**。消费级桌面 app 也不会暴露"让外部编排器驱动我操作这台电脑"的接口。
- **战略上致命**: 把搭子核心能力建在**竞争对手运行时**上 = 把生杀权交给对手 (改 API / 断接入
  / 自捆组织治理层即归零)。护城河不能盖在对手地基上。
- **正确方向是反过来**: **Tandem 当治理 MCP Server**, 它们当 Client —— 它们想碰公司数据/系统/
  动作, 必须连 Tandem, 穿过 4 闸。这是护城河 (路线⑤), 但它治的是"它们的手碰公司数据",
  **不给搭子长出桌面的手**。

### 桌面执行短板的最终取舍

- ❌ **借对手手脚**: 否决 (方向反 + 战略依赖竞品)。
- ✅ **桌面工作盘同步盘** (§四): 自有可控、低风险、契合自用优先。搭子的"会做事" =
  **云端在公司系统内行动 (路线①) + 同步盘把全量工作上下文喂给云端大脑**。
- 本地 OS 自动化那块战场**不进** —— 让员工自己用 Cowork/WorkBuddy 干本地活, 两者在同一台
  电脑各干各的; Tandem 守"组织"这层对手进不来的地方。

### 🅩 再修正 (Owner 2026-06-08): 连"反过来当治理 MCP Server"也降为远期可选

> Owner 2026-06-08: **"我们学习 Cowork/WorkBuddy 的技术和做法, 但它们是竞争产品 —— 接它们的 MCP 没有意思。"**

上面 §六 论证了"反过来 = Tandem 当治理 MCP Server, 它们当 Client 穿 4 闸"。**这条现在也降级**:

- **为什么降**: 竞品产品 (Cowork/WorkBuddy) **不会自愿把自己降级成你网关下的一个 client**。把护城河寄托在"对手愿意配合接入"上 = 没有护城河。与"借对手手脚"对称地不成立。
- **不判生死, 只降优先级**: 整条 MCP/对外互通线 (路线🅩 + B-022 出站 / B-016 Capture / Academy MCP 通道) **挪到远期可选**, 不押近期。
- **保留区分**: 给中央 AI/搭子**自己**装手的**内部**工具回路 (`runToolLoop` 调 `okr.health_digest`/`memory.search`, S1 已落) **不在降级范围** —— 那是自用, 不是对外互通。
- **资源去向**: 全压**自用智能主轴** —— ① 搭子装执行肢体 (内部 tool-loop) + ② B-024 反思 + S2 多步推理。学竞品的**技术做法**吸收进我们自己的栈, 不跟竞品做产品级互通。

## 七、一句话收敛

> Cowork 是"个人雇的、每次重新认识你的、只会动你电脑的超级临时工"。

> 超过的路径不是更能整理本地文件, 是 **会成长 + 有组织效力 + 全量上下文常驻**。
