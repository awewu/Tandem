```
│ 栏 │   [召唤主分身][交付产出][我的OKR]  │ 栏 │

```

**四个实用性问题** (按严重度):

| # | 问题 | 代码出处 | 后果 |
|---|---|---|---|
| 1 | 默认舞台信息密度为零 (一句寒暄) | `TandemPageInner` 默认 `WelcomeStage` | 第一屏不告诉我"今天有什么火要救" |
| 2 | 高频信息被折叠, 召唤栏默认空 | `leftTab/rightTab` 初始 `null` (:185) | 待办/推荐/交付要手点才出, 反模式 |
| 3 | 主舞台与召唤栏数据割裂 | `DeliverCard` 重填 title/body (:457) | "起草→交付"是断的, 非一条流 |
| 4 | 对话被降级成 4 张卡之一 | `CARD_REGISTRY.dialog` | AI 工作台的灵魂被藏起来 |

**判断**: 问题不在视觉 (那套很好), 在**信息架构 + 默认状态**。高频折叠、低频铺满、对话藏起来。

## 2. 设计原则 (本次重设计的不变量)

1. **风格不变**: 只用现有 CHARTER-UI token, 不引入新设计语言。hero-ink / surface-card / pill-* 保留。
2. **三层心智不变**: 你 ↔ 搭子 (Persona) ↔ Tandem AI (中央智囊兜底)。
3. **治理护城河可见**: 区别 WorkBuddy 的关键 —— 搭子的写动作走 zone (红拒/黄 24h 否决/绿留痕),
   布局上要让"代行台账 / 否决窗"是一等公民, 不是埋在审计页。
4. **高频常驻, 低频折叠**: 待办/对话/推荐常驻; 沙盒/技能模式/代表作折叠。
5. **一条流**: 说 → 搭子做 (经治理) → 交付/沉淀, 中途不断点、不重填。

## 3. 目标布局 (方案 C · 驾驶舱 + 对话融合)

### 3.1 桌面默认态 (无 ?card, 无对话)

```
┌──────────────┬────────────────────────────────┬──────────────┐
│ 今日待办 (常驻)│         搭子协作主舞台            │ 产出/交付(常驻)│
│ ───────────  │  ┌──────────────────────────┐  │ ───────────  │
│ 🔴 KR 风险 ×2 │  │ 搭子 · 今天我们干什么?      │  │ 📤 交付       │
│ ✋ 待签字 ×1   │  │ (Persona 名片: 阶段/拿捏分) │  │ 🎨 代表作     │
│ ⏰ 否决窗 ×1   │  └──────────────────────────┘  │ 🧠 我的 Memory│
│ 📜 待复盘 ×3  │                                │              │
│              │   今天先做点什么 (快捷入口)       │ ─ 召唤 (折叠) │
│ ─────────    │   [起决策][起草][对话][代表作]   │ 💬 通用沙盒   │
│ 🤖 搭子升级↑   │                                │ 🛠 技能模式   │
│              │  ┌──────────────────────────┐  │              │
│              │  │ [ 跟搭子说一句话, 立即开干 ] │  │              │
│              │  └──────────────────────────┘  │              │
└──────────────┴────────────────────────────────┴──────────────┘
   ↑ 240px 常驻        ↑ flex-1 max-w-3xl 居中        ↑ 240px 常驻
                                                    ↘ 右下 FAB: Tandem AI
```

**与现状最大差异**:
- 左栏从"48px 空图标条 + 折叠面板"→ **240px 常驻"今日待办"轨道** (吃掉现 InboxCard + RecommendCard 的强信号)。
- 主舞台底部加 **常驻指令框** (对话即主轴, 对位 WorkBuddy)。
- 右栏保留"交付/代表作/Memory"常驻 + "沙盒/技能模式"折叠 (低频)。

### 3.2 对话激活态 (在指令框说话后)

```
┌──────────────┬────────────────────────────────┬──────────────┐
│ 今日待办 (常驻)│  搭子协作流 (对话 + 执行轨迹)      │ 本轮产出      │
│              │  ┌──────────────────────────┐  │ ───────────  │
│ (持续显示,    │  │ 你: 帮我起草 Q3 渠道复盘    │  │ 📄 复盘草稿   │
│  点任意项可    │  │ 搭子: 我查了 3 个 KR...     │  │   (实时填充)   │
│  注入对话上下文)│  │  ┌─ 🔍 okr.read (只读)  ─┐ │  │              │
│              │  │  └────────────────────┘ │  │ [交付到议事室] │
│              │  │ 搭子: 草稿如下 ...         │  │ [存 Memory]   │
│              │  │  ┌─ ✍️ 提议: 填 KR 草稿 ─┐ │  │ [送 IM/邮件]  │
│              │  │  │ 🟡 24h 否决窗 · 待确认 │ │  │              │
│              │  │  └────────────────────┘ │  │ (产出直接带入, │
│              │  └──────────────────────────┘  │  无需重填)     │
│              │  ┌──────────────────────────┐  │              │
│              │  │ [ 继续说... ]              │  │              │
│              │  └──────────────────────────┘  │              │
└──────────────┴────────────────────────────────┴──────────────┘
```

**关键机制 (区别 WorkBuddy 的护城河可见化)**:
- 搭子调只读工具 (okr.read/memory.search) → 轨迹卡显示 `🔍 只读`。
- 搭子提议写动作 → 轨迹卡显示 `🟡 24h 否决窗 · 待确认` (不是 WorkBuddy 那种"直接替你干完")。
- 右栏"本轮产出"**自动带入**主舞台对话产物 → 一键交付, 解决问题 #3 的重填断点。

### 3.3 移动端 (保留现有底部召唤条范式)

## 4. 组件映射 (改什么 / 留什么)

| 现有组件 | 处置 | 说明 |
|---|---|---|
| `SummonRail` (48px 图标条) | **桌面移除**, 移动端保留 | 桌面改 240px 常驻栏 |
| `SummonPanel` (280px 弹层) | 拆分 | 高频内容上常驻栏, 低频留折叠 |
| `InboxCard` | **提到左常驻栏** | 今日待办轨道核心 |
| `RecommendCard` | **并入左常驻栏底部** | 3+1 "请搭子细想" 按钮保留 |
| `DeliverCard` | **提到右常驻栏 + 自动带入对话产出** | 解决重填断点 (#3) |
| `WelcomeStage` | 瘦身 | 保留 Persona 名片 + 快捷入口, 加常驻指令框 |
| `DialogStage` | **升级为主舞台对话流** | 从"跳转/拉 FAB"→ 内嵌对话 + 执行轨迹 |
| `PersonaCard`/`MemoryCard`/`SandboxCard` | 留折叠 | 低频, 折叠在右栏 |
| `useBossAi` (FAB) | 保留 | Tandem AI 中央智囊兜底, 独立于本页对话 |

**新增需要的能力**:
- 主舞台对话流组件 (复用 `/persona` 或 `/chat` 的对话渲染 + `personaActPass` 写提议轨迹卡)。
- 执行轨迹卡: 渲染 tool-loop 的只读调用 + ProxyAction 的 zone/否决窗状态 (数据已存在于 proxy-actions API)。

## 5. 分阶段落地 (降低风险, 可中途验收)

| 阶段 | 内容 | 杠杆 | 状态 |
|---|---|---|---|

## 6. 与 WorkBuddy 的形态差异 (重设计后)

| 维度 | WorkBuddy | 搭子工作台 (重设计后) |
|---|---|---|
| 主轴 | 对话/指令框 | 对话/指令框 ✅ (P3 补齐) |
| 执行可见 | 专家团执行轨迹 | 搭子工具调用 + **zone/否决窗轨迹** ✅ |
| 第一屏 | 指令框 | **今日驾驶舱 + 指令框** ✅ (比 WorkBuddy 多"今日战场") |
| 写动作 | 直接替你干完 | **经 4 闸 + 24h 否决** (护城河) |
| 产出归属 | 个人 | **可一键沉淀组织 Memory / 议事室** (回灌组织 IQ) |

**结论**: 重设计后, 搭子工作台在"对话主轴 + 执行可见"上对齐 WorkBuddy, 同时多出
"今日驾驶舱 + 受治理写动作 + 产出回灌组织"三层 WorkBuddy 结构上给不了的东西。
